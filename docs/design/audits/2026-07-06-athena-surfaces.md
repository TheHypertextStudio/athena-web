## Findings (ordered by severity)

1. **The Athena chat composer is below the fold by default, fully hidden on mobile.** `apps/web/src/app/(app)/orgs/[orgId]/athena/page.tsx:89` sizes the page with `h-full`, but that resolves against the shared content region's total height, not the space left after `RecoveryNudgeBanner` (`apps/web/src/components/app-shell-frame.tsx:253`) — a banner shown by default for every account with zero recovery codes, i.e. every fresh sign-up. Measured: composer `{y:873, height:66}` in a 900px viewport (39px/59% clipped) at 1440×900; `{y:862, height:66}` in an 844px viewport (**entirely off-screen**) at 390×844. Fix direction: give the Athena page's content column its own scroll boundary that accounts for shell chrome (e.g. size against a flex-1 ancestor rather than `h-full`, or move the composer to a `sticky bottom-0` position within the scroll container) so it can never be pushed out by a sibling banner.

2. **A pending approval — the single most important moment in the product — is invisible by default.** After Athena proposes a change, the `ProposalGroupCard` (`ChatProposals` in `athena/page.tsx:225-250`) mounts correctly with real, well-written copy ("Athena proposes 1 change" / "Nothing is applied until you approve"), but the page's auto-scroll effect at `athena/page.tsx:59-61` only depends on `thread?.activities.length`, which doesn't change when the proposals fetch (a separate `useSessionDetail` call) resolves. Verified live: sent "What should I focus on today?", the resulting proposal card was fully present in the DOM (confirmed via `innerText`) but required a manual scroll to see — nothing in the initial viewport hints it exists. Fix direction: re-run the scroll-into-view effect when `proposals.length` transitions from 0, or give the composer form an "N pending" badge that's visible without scrolling.

3. **Raw tool identifiers leak into user-facing copy.** `summarizeToolCall` in `apps/api/src/agent/loop.ts:67-71` falls back to the bare tool name whenever the tool's input has no `title` field — `update_task`'s input is `{state, taskId}`, so every summary for it is the literal string `update_task`. This renders three times in one proposal card (the quiet chip, the row label, and the right-aligned tool column) and once as the sole conversational chip before it. Fix direction: per-tool summary builders (or at minimum a verb+noun template keyed by tool name) so a task-state change reads as "Move '<task>' to In progress," matching the "Searched tasks · 12 results" quality bar the implementation plan itself set.

4. **The Today prompt bar's workspace label is unreadable at mobile width.** `apps/web/src/components/today/today-prompt.tsx:173-203`: the `into <workspace>` label (`truncate text-sm`) shares one un-wrapping flex row with two full buttons ("Ask Athena", "Add task"); at 390px it truncates to **"int…"** — illegible, and worse than simply hiding the label. Fix direction: drop the workspace label at narrow widths (it's already shown in the sidebar header) or stack the label above the button row below a breakpoint.

5. **Two icon-only controls fall below the 40px mobile touch-target minimum.** The recovery-banner dismiss `×` (`recovery-nudge-banner.tsx:101`, `size-8` = 32px) and the mobile search button (`app-shell-frame.tsx:237`, `size-9` = 36px) both render on every authenticated page, including all three surfaces audited here. Fix direction: bump both to `size-10` (40px) or pad the hit target beyond the visual glyph.

Verdict: **BELOW BAR** — dimensions 1, 3, 4, 6, 7, 8 are below the ship bar, and the A11y and No-placeholder gates fail. The empty states (dimension 7's best moments) and dark-mode execution (dimension 5) are genuinely ship-quality and should be preserved as-is; the defects are concentrated in the _populated_ chat state, which is also the state the product's whole pitch depends on.

## Fixes applied (2026-07-06, same day)

All five findings above were fixed and re-verified live against the dev stack (fresh account, real message, no manual scroll) — see `fixed-athena-1440x900.png`, `fixed-athena-390x844.png`, `fixed-athena-proposal-1440x900.png`, `fixed-today-390x844-v2.png`:

1. **Composer visibility** — root cause was `RecoveryNudgeBanner` being rendered as page content (inside `<main>`, alongside `{children}`) rather than shell chrome; any page's `h-full` then measured against `<main>`'s full box while the banner silently consumed part of it. Fixed by giving `AppShell` a dedicated `banner` slot (`packages/ui/src/components/shell/AppShell.tsx`), rendered as a sibling of `<main>` in the shell's own flex column instead of inside the scrollable content (`apps/web/src/components/app-shell-frame.tsx`). Re-measured: composer bottom edge now at y:867 (1440×900, was 939) and y:828 (390×844, was 928) — fully on-screen at both.
2. **Proposal card scroll-into-view** — added a `scrollIntoView` effect in `ChatProposals` (`athena/page.tsx`) keyed on `proposals.length`, independent of the message-list's own scroll effect. Re-verified: after sending a message, the "Athena proposes 1 change" card is in the initial viewport with zero manual scrolling.
3. **Raw tool identifiers** — `summarizeToolCall` in `apps/api/src/agent/loop.ts` now humanizes the tool name (`update_task` → `update task`) for the summary shown in the chip and row label; the small muted `<code>{item.tool}</code>` footnote in `ProposalGroupCard` intentionally still shows the raw identifier (mono, secondary, exactly the rubric's sanctioned use of mono for IDs) — that one was never the violation.
4. **Mobile prompt-bar truncation** — `today-prompt.tsx`'s action row now wraps the "into `<workspace>`" label onto its own full-width line below the buttons at narrow container widths (`order-2 basis-full`, resetting to the original single-row layout at `@lg`), instead of squeezing it to an unreadable `"int…"`.
5. **Touch targets** — bumped all four `size-8`/`size-9` icon-only shell buttons (recovery-banner dismiss, mobile hamburger, mobile rail trigger, mobile search) to `size-10` (40px) for consistency, not just the two originally cited.

Full validation after the fixes: `packages/ui` (255/255), `apps/web` (229/229), `apps/api` (1091/1091) tests pass; typecheck and lint clean for all three touched packages.

## Craft pass: transitions, motion, responsiveness, typography (2026-07-06, same day)

A follow-on review targeted the dimensions the first pass didn't probe deeply. Verified live against the running dev stack (instrumented `document.startViewTransition` to count real calls, not just code presence).

- **Shared-element transitions — was below bar, now fixed.** Every ghost row already carried a stable `view-transition-name` (`proposal-${activityId}`), but nothing ever called `startViewTransition` — measured 0 calls across a full ask → propose → approve cycle. Fixed at the shared root cause: `apps/web/src/lib/use-session-detail.ts`'s `load()` (used by both the session-detail page and the Athena panel/page's `ChatProposals`), `apps/web/src/components/athena/athena-conversation.tsx`'s `load()`, and `apps/web/src/components/today/ghost-proposals.tsx`'s `load()` now commit their post-fetch state inside the app's existing `startViewTransition` helper (`apps/web/src/lib/view-transition.ts`, the same helper already used correctly by the agenda view). Re-verified live: instrumented count went from 2 → 3 (exactly one increment) across a real approve click.
- **Animations/motion — crafted.** Motion tokens (`--dur-fast/base/slow`, `--ease-out/in-out`) used correctly throughout the new/touched files; the ⌘J panel correctly inherits the Sheet primitive's tokened slide-in (measured `transitionDuration: 0.24s` live); `prefers-reduced-motion: reduce` collapses it to near-instant and the panel stays fully usable.
- **Layout responsiveness — crafted**, with one stray viewport-breakpoint fixed for consistency: `today-prompt.tsx`'s `⌘↵` keyboard hint used `sm:inline` (a viewport breakpoint) next to container-query classes on the same row; changed to `@lg:inline` to match the row's own container-query vocabulary.
- **Typography — two real misses, both fixed.** `ProposalGroupCard`'s card title (`apps/web/src/components/agents/proposal-group-card.tsx`) used `text-body font-medium` for what the rubric defines as a `text-h3` element (a card title) — corrected. `SheetTitle` (`packages/ui/src/primitives/sheet.tsx`) hardcoded `text-base font-semibold` (not a named token), and the new Athena panel's `asChild` title span duplicated it with `text-sm font-semibold` — two typography classes on one element, whichever wins by CSS build order rather than anything visible in the JSX. Fixed by giving `SheetTitle` itself the named `text-h3` token and removing the duplicate override.

All four fixes verified live (screenshots + the instrumented view-transition count) and by a full re-run of `packages/ui` (255/255) and `apps/web` (229/229) tests plus a full-repo typecheck/lint (12/12 packages clean).

**Still not done**: the "add a connector" step doesn't yet show up as a proactive nudge inside a conversation (e.g. Athena suggesting "connect Sunsama" when asked about it) — today it's a manually-discovered button, not a contextual prompt. ## Separate chat sessions (2026-07-06, same day)

The original goal explicitly asked for storage to support "separate sessions" without needing a full past-chat browser yet — this had been overlooked. Checked `getOrCreateChatSession` (`apps/api/src/routes/agent-sessions.ts`) and found it was architecturally locked to exactly one `kind: 'chat'` session per org, always resumed by "newest wins" with no way to ever start a second one.

Added `POST /v1/orgs/:orgId/sessions/chat/new` — creates a genuinely new `kind: 'chat'` session without deleting or touching the prior one. Because `GET /chat` and `POST /chat/messages` both already resume the newest chat session, every door onto "the" thread (⌘J panel, standalone page) automatically continues from the fresh session the moment it's created — no other route needed to change. Older sessions stay fully queryable (`GET /:id`, the Agents feed) — there is deliberately no dedicated history browser, matching "don't make too big a deal of it yet."

Web: a "New chat" button next to "Connect a tool" in `AthenaConversation`, disabled when the current thread is already empty. Verified live: sent a real message, clicked New chat, watched the thread clear via the same view-transition used for ghost approvals, confirmed via API response the new session has a different id and zero activities, and confirmed the _old_ session's message was still present when read directly. Added a new API test (`apps/api/tests/routes/agent-chat.test.ts`) asserting the same: new session id ≠ old, old history intact, `GET /chat` resumes the new one. Full re-run: `apps/api` 1092/1092 tests, full-repo typecheck + lint 12/12 packages, both clean.
