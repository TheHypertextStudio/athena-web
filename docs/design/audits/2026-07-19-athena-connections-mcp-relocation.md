## Findings resolved in the first pass

1. **Google Tasks account row wrapped into an unreadable vertical stack at narrow widths** — `apps/web/src/components/settings/gtasks-accounts-section.tsx` (`GtasksAccountRow`). The scope-label span (`"Two-way · all lists"`) had no truncation, and its `shrink-0` action-button sibling squeezed the label column toward zero width. Fixed with the same `flex-wrap ... sm:flex-nowrap` pattern already proven in `integration-provider-card.tsx`, plus `truncate` on the scope label.
2. **`McpConnectorsSection`'s own header row had the identical bug**, newly prominent once it moved onto the Athena page — `apps/web/src/components/settings/mcp-connectors-section.tsx`. "Tools for Athena" + its description got squeezed by the non-wrapping "Add connector" button at 390px. Fixed with the same `flex-wrap items-start gap-4 sm:flex-nowrap sm:justify-between` pattern; re-verified by screenshot.

## Findings resolved in the follow-up pass

3. **Misdiagnosed as an app-shell bug; actually Next.js's dev-mode indicator.** Disabled with `devIndicators: false` in `apps/web/next.config.ts` — dev-only chrome that never shipped to production, but it was cluttering local design-review captures and reading as a real defect. Re-verified absent from all 8 shots in this pass.
4. **Terse "No tools yet." empty state** — `apps/web/src/components/settings/mcp-connectors-section.tsx`. Replaced with an icon + explanation pattern matching the Google Tasks empty state: "No tools connected yet. Add a connector so Athena can act through the services you use." (branches to an admin-facing variant when `canManage` is false).
5. **No visual boundary between identity linking and workspace connections on the Connections page** — `apps/web/src/components/settings/connected-accounts-tab.tsx`. Added a `"Linked accounts"` label above the Google identity card, matching the label convention every other subsection on the page already used, so the page now reads as one consistent list of labelled sections instead of two conventions.

## Findings resolved in the tonal-hierarchy pass

Direct user feedback: cards and rows across both pages were stacking `border-outline-variant ... border` (and, in one place, a dashed border) on top of a `bg-surface-container-low` tint that was already doing the visual-separation work on its own — over-relying on lines where MD3's surface/surface-container tonal ramp should carry the hierarchy instead. Fixed across every card, row, empty state, and disclosure panel touched this session:

- `gtasks-accounts-section.tsx` — the account row card, the identity-picker panel and its empty state, and the "no connections" empty state all dropped their borders; the row's inline error/feedback footers moved from a `border-t` divider to a `bg-surface-container` tonal step.
- `mcp-connectors-section.tsx` — the connector row card, the "Advanced options"/"Other connection methods" disclosure panels in the add-connector form, and the empty state all dropped their borders.
- `integration-provider-card.tsx` and `provider-group.tsx` — the same outer-card border removed, and `integration-provider-card.tsx`'s error/pending footer note moved to the same `bg-surface-container` tonal-step pattern.
- `apps/web/src/app/(app)/settings/athena/page.tsx` — the "Working preferences" card had a border with **no** tonal fill at all (the one clearest violation of the two); it now uses `bg-surface-container-low` alone.

Left as-is, deliberately: form inputs/selects keep their border (a genuine interactive-affordance case, not a separator), and the hairline `divide-y` between individual linked-account rows inside one already-flat `ProviderGroup` card (a standard MD3 list-divider use, not "everything"). The shared `SectionHeader`'s `border-b` under every Settings page title was also left alone — it's shared infrastructure used across dozens of unaudited pages, out of proportion for a scoped cleanup without screenshot-verifying the rest of Settings.

## Findings resolved in the MCP-terminology pass

Direct user feedback: the section never said "MCP" anywhere a sighted user could see it — only in code comments and a screen-reader-only `aria-label`. The plain-language "Tools for Athena" framing is deliberate for a casual user, but a technical user who already knows what MCP is (or is following external MCP-related documentation) had no textual anchor confirming this was the right place. Fixed in `mcp-connectors-section.tsx`:

- Added a quiet one-line caption under the section description: "Built on MCP (Model Context Protocol), the open standard for AI tool access."
- The add-connector dialog description now reads "Give Athena access to a remote MCP (Model Context Protocol) server it can use as a tool," instead of the fully de-jargoned "Give Athena access to a service you use."

Both are secondary to the primary plain-language copy (smaller/muted caption, a dialog description rather than the heading) so a non-technical user isn't confronted with the acronym up front, but it's discoverable by scanning or search either way. Re-verified by screenshot at 1440×900 and 390×844 — the added line wraps as normal prose, no layout regression.

Verdict: **SHIP BAR** — every dimension ≥3, every gate green, zero open findings.
