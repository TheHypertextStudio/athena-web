## data-model

### §0.1: "Decision ULID vs cuid2 is an open issue; pick one and never mix."

**Decision.** ULID, repo-wide, single generator. `@docket/db/src/id.ts` exports `genId = () => ulid()` (from the `ulid` npm package), used as `.$defaultFn(genId)` on every app-table `id` column AND wired into Better Auth `advanced.database.generateId: () => genId()` so auth-minted rows (user/session/account/...) use the same 26-char text ULID. No `uuid()`, no cuid2, no mixing. Add `ulid` to `@docket/db` deps; pin a version in the root catalog.

**Why.** RECONCILIATION line 7 and the engineering plan line 216 both fix this as a tie-breaker (ULID text, one generator) precisely so `actor.user_id` FKs line up with Better Auth's `text("id")` PK. The data-model note is the only place still calling it 'open'; it is in fact resolved. ULID is sortable (good for created-order indexes) and matches Better Auth's text-id default shape.

### Deferred verification: does Better Auth 1.6.14 honor `advanced.database.generateId` so app + auth rows share one ULID generator? What is the fallback if not?

**Decision.** VERIFIED present and used: set `advanced: { database: { generateId: () => genId() } }` in `@docket/auth`. The Drizzle adapter is configured WITHOUT `usePlural` (singular table names: `user`, `session`, `account`, `verification`) to match the data-model schema; do NOT set `useNumberId` (it is global and would force serial ints, breaking the text-ULID contract). Better Auth CLI generates schema into `packages/db/src/schema/auth.ts`. Fallback (only if a future BA version drops the global hook): keep `generateId` returning ULID per-call; if it is ever ignored for a specific model, add a Drizzle `$defaultFn(genId)` override on that generated column after CLI generation (re-applied via a post-generate codemod in `pnpm db:auth:generate`), so the text-ULID invariant holds regardless. Never let any auth table fall back to `crypto.randomUUID()` (would fracture the FK graph).

**Why.** Context7 current Better Auth docs confirm `advanced.database.generateId: (options) => string | false` is a supported, non-deprecated option (the docs even show per-model branching), so the data-model's planned wiring works as written. The `usePlural`/`useNumberId` cautions come from the same docs and protect the text-singular contract the schema depends on. The codemod fallback keeps zero builder judgment if internals shift.

### §6.6 label: "(open issue: per-org vs per-team uniqueness)" on `label_org_name_uq`.

**Decision.** Uniqueness is PER-ORG-PLUS-SCOPE, not globally per-org. Replace `uniqueIndex("label_org_name_uq").on(organizationId, name)` with a partial-pair scheme: (a) `uniqueIndex("label_org_name_global_uq").on(organizationId, name).where(sql\`team_id IS NULL\`)`for org-global labels, and (b)`uniqueIndex("label_team_name_uq").on(organizationId, teamId, name).where(sql\`team_id IS NOT NULL\`)`for team-scoped labels. This lets a team have a label of the same name as an org-global one and lets two teams each have their own 'Bug' label, while preventing duplicates within the same scope.`name`comparison is case-sensitive as stored (UI may normalize).`color`is a`#RRGGBB` 7-char hex (app/Zod validated, not DB-checked).

**Why.** The data-model already gives `label` an optional `team_id` scope (line 505), so uniqueness must be scope-aware or the optional scope is meaningless. Per-org-global + per-team is the least surprising for a Linear-style product where teams own their own label sets but org-wide labels also exist. Two partial unique indexes encode this with zero ambiguity for the builder.

### RECONCILIATION #5 vs data-model: add `visibility {public,private}` and `ancestor_path text[]` (GIN) to team/program/project/task — the data-model only declares the enum, not the columns.

**Why.** RECONCILIATION line 11 mandates these columns and the permissions resolver (§7.2, §7.3) reads `ancestor_path @> ARRAY['project:X']` for GIN-indexed cascade closure and reads the per-row `visibility` for the member-sees-public default. ### GAP: RECONCILIATION (line 28) requires a Postgres `idempotency_key` table (24h TTL) for v1, but the data-model spec never defines it.

**Decision.** Add `idempotency_key` to a new `schema/infra.ts` (or `crosscutting.ts`): `key text NOT NULL; organization_id text →organization CASCADE (nullable — cross-org Hub/admin POSTs have no org); user_id text NOT NULL →user CASCADE; method text NOT NULL; path text NOT NULL; request_hash text NOT NULL; response_status integer; response_body jsonb $type<unknown>; status idempotency_status {in_progress,completed} NOT NULL DEFAULT 'in_progress'; created_at timestamptz DEFAULT now() NOT NULL; expires_at timestamptz NOT NULL`. A lightweight cron (reuse the lifecycle sweep cadence) hard-deletes rows where `expires_at < now()`. `request_hash` (sha256 of method+path+canonical body) lets the middleware 409 a key-reused-with-different-body. **Why.** RECONCILIATION line 28 pins 'a Postgres idempotency_key table (24h TTL) for v1 — no Redis', and the api contract §1.7/§3 says all POST create routes accept an `Idempotency-Key` header 'validated, stored 24h'. The data-model is the single SQL owner so the table must live here. User-scoped PK + request_hash is the standard correct shape (matches Stripe's idempotency semantics: same key + different body = error).

### Direction/semantics of `actor.user_id` FK cascade vs the engineering rule 'User is a global account that persists at zero memberships.'

**Decision.** Keep `actor.user_id text →user.id ON DELETE CASCADE` exactly as written, but document the directionality explicitly: the cascade fires only when a USER is hard-deleted (then all their actor membership rows go too) — it never fires the other way. 'Leaving an org' = deleting the `actor` row (or archiving it), which does NOT touch `user`. Account deletion is the only path that removes a user; org lifecycle purge deletes actors via `organization_id` CASCADE, not via `user_id`. The partial unique index `actor_org_user_uq ON (organization_id, user_id) WHERE user_id IS NOT NULL` stays (one human membership per user per org; agent/team actors have null user_id and are exempt). No schema change.

**Why.** Engineering line 70 says the User survives at zero memberships and 'removing them from an org removes memberships, not the account.' The FK as written (actor→user CASCADE) is correct for that: deleting a user removes their memberships (desired), while removing a membership is an `actor`-row delete that leaves `user` intact. The only ambiguity was directional; pinning it removes builder doubt. The partial unique index is already in the spec and is correct.

## api-rpc-contract

### §1.7 IDs: 'branded string types per entity (OrganizationId, TaskId, …) — opaque to clients.' Are per-entity branded id schemas authored, or the single shared Id from RECONCILIATION #7/#13?

**Decision.** Author ONE shared branded ULID primitive `Id = z.string().regex(/^[0-9A-HJKMNP-TV-Z]{26}$/).brand('Id')` in `@docket/types/primitives.ts`. Per-entity ids (`OrganizationId`, `TaskId`, etc.) are TYPE-LEVEL brands only — `export type TaskId = Id & z.BRAND<'TaskId'>` plus an `id<'TaskId'>()` factory that returns the SAME runtime Zod schema (ULID validation) with an added phantom brand for compile-time safety. The wire/runtime validator is identical for every id (26-char Crockford ULID); only the TS type differs. MCP's `Id = z.string().uuid()` (mcp-surface §3.3) is OVERRIDDEN to this exact schema. Branded ids are opaque strings to clients; no UUID anywhere.

**Why.** RECONCILIATION #1 (ULID repo-wide, no UUID), #7 (one branded Id shared by API + MCP, MCP must not use uuid), and #13 (Id primitive identical across surfaces) are the tie-breaker. Per-entity compile-time brands give the developer-experience the §1.7 spec wants without two divergent runtime validators. genId in `@docket/db/src/id.ts` produces the ULID these validate.

### §1.4 / RECONCILIATION #3: RoleCreate and GET /orgs/:orgId/roles use `capabilities: CapabilityGrid` (a per-resource `Record<ResourceType, Capability[]>` grid). RECONCILIATION mandates a FLAT `GrantCapability[]`. Which wins?

**Decision.** FLAT wins. `Role.capabilities`, `RoleCreate.capabilities`, and `RoleUpdate.capabilities` are `Capability[]` (the bare `z.enum(['view','comment','contribute','assign','manage'])` array), resolved by max-rank per permissions §1. DELETE `CapabilityGrid` from the §3.1 type sketches and from the `@docket/types` inventory (§5). The `GET /orgs/:orgId/roles` output becomes `Role[]{ id, name, key, isSystem, baseCapability, defaultVisibility, capabilities: Capability[] }`. The per-resource grid concept does NOT exist; per-resource scoping is expressed via `Grant` rows (resourceKind + resourceId + capabilities[]), not via a role grid.

**Why.** RECONCILIATION #3 and #6 are explicit: Role.capabilities is a flat GrantCapability[] jsonb, the bare literal is the stored/Zod value, and the API RoleCreate DTO 'aligns to the flat shape.' The data-model role table (§6.1) stores `capabilities: jsonb GrantCapability[]`. CapabilityGrid contradicts all three; removing it is the only consistent path.

### §3.7 task routes expose `POST /:taskId/links` returning `ExternalLink` and `GET /:taskId` returning `externalLinks: ExternalLink[]` (a one-to-MANY external-link relationship). The data-model has NO external_link table — provenance is a SINGLE inline triple on `task` (source_integration_id, external_id, external_url, sync_mode) with `task_source_uq`. How is link_external/unlink_external modeled?

**Decision.** v1 ships SINGLE-provenance, not a link collection. `GET /:taskId` returns `provenance: { source: 'native'|'linked', sourceIntegrationId?, externalId?, externalUrl?, syncMode? }` (the inline triple) and `externalLinks` is REMOVED from TaskOut for v1. `POST /:taskId/links` is renamed to `POST /:taskId/link-external` taking `{ integrationId, externalId, externalUrl, role }`; it SETS the task's inline provenance fields (source='linked', sync_mode='mirror') and returns the updated `TaskOut`. It 409s (`ConflictError`, code `task_already_linked`) if the task is already linked to a different external resource (one provenance per task, enforced by `task_source_uq`). `DELETE /:taskId/links/:linkId` is renamed `DELETE /:taskId/link-external` (no linkId param) and CLEARS the inline provenance (source back to 'native'). The MCP `link_external` tool (mcp-surface §3.4) maps to the same handler. A true multi-link table is deferred to a fast-follow; if added later it is additive (new `task_external_link` table) and does not change these v1 routes.

**Why.** The data-model is authoritative on schema (RECONCILIATION preamble points to data-model names as authoritative; build-sequence Part 0 says data-model §5 governs tables). There is exactly one provenance triple per task and a unique index enforcing it, so a `ExternalLink[]` collection has no backing table in v1. Renaming to singular link-external and clearing on unlink matches the inline model and the §1.7 'linked rows read-only except link_external/unlink_external' rule. Keeping it additive preserves forward-compat without inventing a table.

### §1.7 'Idempotency: all POST create routes accept optional Idempotency-Key header (validated, stored 24h).' What store, what key format, what is the replay contract, and which routes exactly?

**Decision.** Store = a Postgres `idempotency_key` table (RECONCILIATION: 'a Postgres idempotency_key table (24h TTL) for v1 — no Redis'). Schema: `{ key text, organization_id text null, user_id text not null, route text not null, request_hash text not null, response_status int, response_body jsonb, created_at timestamptz default now(), PRIMARY KEY (user_id, route, key) }`, with a TTL sweep (the same `CRON_SECRET`-guarded cron) deleting rows older than 24h. Header schema: `Idempotency-Key = z.string().uuid()` (UUID v4; clients generate it — this is a CLIENT-supplied request token, NOT an entity id, so UUID is correct here and does not violate the ULID rule which governs stored entity ids). Replay contract: on a POST with a key, look up `(user_id, route, key)`; if found AND `request_hash` matches → return the stored `response_status`+`response_body` verbatim (no re-execution); if found AND hash differs → 422 `Problem` code `idempotency_key_reuse`; if not found → execute in the same tx that inserts the row, storing the final response. Applies to EVERY `POST` that creates a row: `POST /orgs`, members/invite, roles, `POST /…/teams|initiatives|programs|projects|milestones|cycles|tasks|comments|updates|agents|sessions|integrations`, task subroutes that create (`/dependencies`, `/link-external`), `POST /dailyplan/items`. It is OPTIONAL (absent header → normal execution). The MCP `idempotency_key` tool arg (mcp-surface §3.1) maps to the same table and contract.

**Why.** RECONCILIATION explicitly resolves the store (Postgres table, 24h, no Redis). The header carries a client-generated request token (RFC-style idempotency), which is conventionally a UUID and is distinct from entity ids — so UUID here is consistent with RECONCILIATION #1/#7 (those govern entity/db ids). Storing status+body+hash gives exactly-once-semantics on retries and detects key-reuse-with-different-body, the standard correctness contract.

### §3.11 sessions router is mounted `/orgs/:orgId/sessions` but the backing table is `agent_session` (deliberately NOT `session`, which is Better Auth's login session). Does the route path / RPC namespace collide, and what is the canonical naming?

**Decision.** No collision — keep the REST path `/orgs/:orgId/sessions` (these are agent sessions, org-scoped, under `/v1`; Better Auth's login `session` lives under `/api/auth/*` which is OUTSIDE `/v1` and NOT in AppType per §1.6/§4). The output DTO is `SessionOut` mapping `agent_session` columns. No path or type collision exists because the two live in disjoint URL spaces (`/v1/orgs/:orgId/sessions` vs `/api/auth/*`).

**Why.** data-model §5.6 note and build-sequence Part 0 both flag the `session` name as reserved for Better Auth. ### §3.11 `POST /:sessionId/approvals/:activityId` requires `org:assign` with 'default approver = task's assigner/delegator; configurable per Org/Team via approval_routing.' RECONCILIATION defers 'approval-routing storage' (Agent column vs Team settings). For the API contract, where is approval_routing read from, and what is the v1 resolver?

**Decision.** Storage = the `agent.approval_routing` jsonb column (it already exists in data-model §5.5: `approvalRouting: jsonb $type<ApprovalRouting>() // org/team override of approver=assigner`). For v1, `ApprovalRouting = { defaultApproverActorId?: ActorId, byTeam?: Record<TeamId, ActorId | { teamId: TeamId }> }`. The approver resolver (permissions §9.2) runs in this fixed order: (a) `agent.approval_routing.byTeam[task.teamId]` if set, else `agent.approval_routing.defaultApproverActorId` if set; (b) the task's assigner/delegator (the actor who set assignee_id/delegate_id to this agent — derived from the latest `audit_event` of type `assigned`); (c) `agent.accountableOwnerId`; (d) org Owners. Each candidate MUST also pass `canActor(approver, 'assign', target)`, else fall through. The Org-level default is represented as the agent's `defaultApproverActorId` (no separate org column in v1). `POST /:sessionId/approvals/:activityId` 403s if the caller is not in the resolved approver set OR lacks `assign` on the target. A separate per-Org approval-routing table is deferred; the jsonb column is forward-compatible.

**Why.** data-model already provisions `agent.approval_routing` jsonb as the override store, so v1 needs no new table — resolving RECONCILIATION's deferred 'Agent column vs Team settings' in favor of the Agent column that already exists. The resolver order is taken verbatim from permissions §9.2 (the authoritative routing logic), and the `assign`-on-target filter is mandated there, keeping the API contract and permissions engine in lockstep.

### §3.11 `GET /:sessionId/stream` is SSE (text/event-stream). RECONCILIATION pins 'Session live transport: SSE for v1; one mechanism.' Confirm the transport, framing, resume semantics, and how it coexists with the RPC AppType (SSE responses break hc<AppType> typing).

**Decision.** SSE confirmed as the ONLY live transport for v1 (no WebSocket). Implement with Hono's `streamSSE` helper at `GET /v1/orgs/:orgId/sessions/:sessionId/stream`. Framing: each event is `{ id: <session_activity.id (ULID, monotonic)>, event: 'activity'|'status'|'ping', data: <JSON of SessionActivityOut | {status} > }`. Resume: client sends `Last-Event-ID` header (or `?since=<activityId>` query, both honored); the server replays `session_activity` rows with `id > since` then tails live. Heartbeat `event: ping` every 25s to keep the connection alive on Vercel. RPC coexistence: the streaming endpoint is EXCLUDED from the typed RPC surface — it is consumed via the browser `EventSource`/`fetch`-stream directly (NOT via `hc<AppType>`), and its `describeRoute` declares `responses['200'].content['text/event-stream']` for OpenAPI only. The route still lives on the chained router so it appears in the spec, but the client wrapper provides a hand-typed `streamSession(orgId, sessionId, { since })` helper returning an async iterator of `SessionActivityOut`. This is the same SSE mechanism MCP resource subscriptions reuse (mcp-surface §4.4) — one fan-out from the service-layer event bus.

**Why.** RECONCILIATION pins SSE as the single v1 mechanism. Hono's streamSSE is the supported primitive. SSE bodies are not JSON so they cannot ride the RPC inference (which expects `.json()`); excluding the stream from hc typing while keeping it in the OpenAPI doc preserves both the §6 'one rule' (validator-based RPC) for all JSON routes and a documented stream endpoint. Replay-by-id and 25s heartbeats are the standard SSE-on-serverless reliability pattern, and reusing the service event bus matches mcp-surface §4.4's fan-out.

### §1.5/§2 security scheme `mcpOAuth` and the per-route `x-docket-capability` extension: how do the `org:<cap>` route labels (annotation only per RECONCILIATION #6) map to the runtime capabilityGuard, and how do MCP scopes layer on top?

**Decision.** `x-docket-capability` is an OpenAPI vendor extension carrying a string from the closed set `{public, authenticated, org:view, org:comment, org:contribute, org:assign, org:manage, staff:support, staff:finance, staff:superadmin}`. It is DOCUMENTATION + a runtime assertion source ONLY — it is NOT the stored capability. The `capabilityGuard(required, locate)` middleware (§1.6) parses `org:<cap>` → the bare `Capability` literal (`org:contribute` → `'contribute'`) and calls `canActor(actorId, cap, target)` from `@docket/authz` (permissions §6). The bare literal is what's stored/compared (RECONCILIATION #6). MCP scope layering: for the `/mcp` surface (not the RPC routes), the token scope (`work:read|work:write|agents:run|connectors:link`) is a coarse gate checked BEFORE `canActor` — both must pass (permissions §10). The mapping scope→capability-class is mcp-surface §2.2. On the RPC routes there is no scope gate (session-cookie auth), only `canActor`. The `bearerAuth` scheme covers RPC bearer/cookie auth; `mcpOAuth` covers only `/mcp` (which is outside AppType). A route's `security` array lists `[{ bearerAuth: [] }]` for RPC routes; `/mcp` declares `mcpOAuth` with its scopes.

**Why.** RECONCILIATION #6 makes `org:*` an annotation label only and the bare literal the stored value — so the guard must translate label→literal and delegate to the one `canActor` engine (permissions §10: 'one engine, three call sites'). permissions §2.2/§10 define the two-layer scope+grant model for MCP exclusively, so layering scopes onto RPC routes would be wrong; cookie-authed RPC routes use canActor alone.

### RECONCILIATION (deferred MCP): 'verify Better Auth 1.6.14 emits client_id_metadata_document_supported (CIMD), honors RFC 8707 resource→aud stamping, and surfaces getMcpSession().scopes. If any is absent, add a thin RS shim.' Decide the concrete fallback/shim plan for the API/RS.

**Decision.** Decision per capability (verified against Better Auth source via ctx7: `mcpHandler`/`verifyAccessToken` DO take an `audience` option and `handleMcpErrors(error, audience, opts)` emits WWW-Authenticate with resource_metadata; `getMcpSession`/`withMcpAuth` exist). (1) AUDIENCE/RFC 8707: rely on Better Auth's `verifyAccessToken({ verifyOptions: { audience: MCP_RESOURCE_URL } })` to reject `aud` mismatches. FALLBACK if `aud` is not stamped from the `resource` param: the RS asserts audience itself by decoding the token claims and requiring `aud === MCP_RESOURCE_URL` (or, if `aud` absent entirely, bind on `iss === MCP_ISSUER_URL` + a hardcoded single-resource allowlist) — a ~20-line `assertAudience()` in `apps/api/src/mcp/auth.ts`. (2) CIMD `client_id_metadata_document_supported`: if Better Auth's AS metadata does not include the flag, the RS OVERRIDES the discovery doc — serve `/.well-known/oauth-authorization-server` via a wrapper that spreads Better Auth's `oAuthDiscoveryMetadata(auth)` output and force-sets `client_id_metadata_document_supported: true` plus a CIMD-resolution hook (fetch client_id URL, validate client_id===URL, SSRF/domain allowlist) implemented in the RS. DCR (`/register`) stays as the MAY fallback (Better Auth provides it). (3) `getMcpSession().scopes`: if `scopes` is missing from the returned object, the shim reads scopes from the verified token's `scope` claim (space-delimited) directly. ALL three shims live in `apps/api/src/mcp/auth.ts`, are unit-tested against a stubbed AS, and are guarded by a boot-time assertion `pnpm env:check`-adjacent probe that fetches the live AS metadata and logs which path (native vs shim) is active. No behavior is left unverified at runtime.

**Why.** ctx7 confirmed Better Auth's MCP layer already does audience-bound verification and WWW-Authenticate emission, so the primary path is native; the spec's 'add a thin RS shim if absent' is honored by concrete, located, tested overrides for each of the three capabilities. mcp-surface §2.5/§2.6 already specify the exact audience/issuer/scope checks and challenge forms, so the shims simply guarantee those invariants regardless of plugin completeness. RECONCILIATION explicitly authorizes a net-new RS shim.

### §3.16 admin routes: `GET /admin/audit` requires `staff:superadmin` but most read routes are `staff:support` — and impersonation runs canActor as the impersonated Actor (permissions §11). How does the admin router resolve the staff principal vs the org middleware, and what auth wiring applies (admin routes are not org-scoped)?

**Decision.** Admin routes are a SEPARATE authorization plane (permissions §11) and do NOT use `orgContextMiddleware`/`capabilityGuard`. The `/admin` router applies, in order: (1) `sessionMiddleware` (must have a valid Better Auth session), (2) `staffGuard(minRole)` which loads the `StaffUser` row for `session.user.id` and asserts `staffRole >= minRole` using the ordering `support < finance < superadmin` (support=1, finance=2, superadmin=3), 403 otherwise. Per-route `x-docket-capability` values `staff:support|staff:finance|staff:superadmin` map directly to `staffGuard('support'|'finance'|'superadmin')`. Note: `staff:finance` is NOT >= `staff:support` under a strict linear order (finance and support are different duties), so the guard checks membership in an explicit allow-SET per route, NOT a single chain: support routes allow `{support, finance, superadmin}`? — DECISION: use a strict numeric ladder where superadmin ⊇ finance ⊇ support is FALSE; instead each route declares the EXACT roles permitted. Canonical mapping: read/metrics/users/orgs/lifecycle/impersonation = `{support, finance, superadmin}`; billing-write/holds = `{finance, superadmin}`; audit/staff-management = `{superadmin}`. Impersonation: `POST /admin/impersonations` mints an `ImpersonationSession`; subsequent TENANT calls made while impersonating still run through `canActor` as the impersonated Actor (operators get NO tenant super-grant), and every admin action writes an `OperatorAuditEvent` (never the tenant `audit_event`). Every `/admin/*` mutation writes `OperatorAuditEvent` (§3.16). Admin is consumed by `apps/admin` via the same RPC AppType (the `/admin` group is part of the chained app).

**Why.** permissions §11 mandates the operator plane be wholly separate from tenant canActor, impersonation be scoped/time-boxed/audited and still run as the impersonated actor, and StaffUser never appear in tenant grants. The data-model staffRole enum is `{support, finance, superadmin}` with no implied total order, so a per-route allow-set (rather than a single ladder) is the only correct interpretation of the §3.16 capability column — superadmin alone sees audit/staff, finance+superadmin do billing, all three do reads.

### §3.13/§3.14/§3.19 cross-org routes (notifications, dailyplan, hub) are `authenticated` with downstream per-resource `org:view`/`org:assign`. How is the org context resolved when there is no `:orgId` path param, and how is per-item capability filtering enforced without orgContextMiddleware?

**Decision.** Cross-org routes (`/notifications`, `/dailyplan`, `/hub/*`) do NOT run `orgContextMiddleware` (no `:orgId` in path). They run `sessionMiddleware` only, then resolve the caller's set of active human `Actor` rows across ALL orgs in one query (`actor WHERE user_id = session.user.id AND kind='human' AND status='active'`). Aggregation = fan-out per membership, server-merged, NEVER a cross-tenant SQL join (§3.15). Each returned item carries `organizationId` and is individually filtered by calling `canActor(actorForThatOrg, 'view', item)` via the query-scoping predicate (permissions §7.2) composed per-org — a guest never sees hidden work even in an aggregate. For the downstream-mutating cross-org routes: `POST /notifications/:id/act` and `POST /dailyplan/items` resolve the specific org from the row/payload (`notification.organization_id`; `dailyplan item taskRef.organizationId`), load the caller's Actor in THAT org, then enforce `org:assign` (notifications act) / `org:view` (dailyplan add) via `canActor`. If the caller has no active Actor in the referenced org → 404 (existence-hiding, permissions §7.5). DailyPlanItem rows carry NO organization_id (data-model §6.3: Hub is cross-org, org via `ref_organization_id`); ownership is the Hub (1:1 with user), so item owner-checks compare `dailyPlanItem.hubId` to the caller's hub.

**Why.** §3.13–§3.15 explicitly state these aggregate across the caller's active Actors, carry per-row organizationId, fan-out (no cross-tenant join), and per-item capability-filter. permissions §7.2 provides the query-scoping predicate and §7.5 the 404 existence-hiding. data-model §6.3 confirms dailyPlanItem has hubId (owner) + ref_organization_id (chip) and no organization_id, so owner checks key on hubId and downstream capability checks key on ref_organization_id.

### §1.2/§3 'Hono router split across many files; method-chaining required; org-scoped resources mounted under /orgs/:orgId via orgs.route(...).' Confirm the exact composition so RPC inference survives, including where cross-org groups and the auth/mcp mounts sit relative to AppType.

**Decision.** Canonical composition: each resource group is its own chained `new Hono<AppEnv>()` in `apps/api/src/routes/<group>.ts`, NEVER breaking the chain (no mid-route const). The org-scoped groups are chained onto a single `orgs` router: `const orgs = new Hono<AppEnv>().get('/', …).post('/', …).get('/:orgId', …).route('/:orgId/teams', teams).route('/:orgId/initiatives', initiatives).route('/:orgId/programs', programs).route('/:orgId/projects', projects).route('/:orgId/cycles', cycles).route('/:orgId/tasks', tasks).route('/:orgId/comments', comments).route('/:orgId/updates', updates).route('/:orgId/agents', agents).route('/:orgId/sessions', agentSessions).route('/:orgId/integrations', integrations).route('/:orgId/members'…(folded into orgs)…).route('/:orgId/roles'…).route('/:orgId/grants'…).route('/:orgId/billing'…)`. The root app: `const app = new Hono<AppEnv>().basePath('/v1'); const routes = app.route('/orgs', orgs).route('/notifications', notifications).route('/dailyplan', dailyplan).route('/hub', hub).route('/admin', admin).route('/integrations/oauth', oauthCallbacks); export type AppType = typeof routes;`. The `orgContextMiddleware` is applied INSIDE the `orgs` router as `orgs.use('/:orgId/*', orgContextMiddleware)` BEFORE the child `.route()` chains so child handlers see `c.var.actorCtx`. `/api/auth/*`, `/mcp`, `/.well-known/*`, `GET /v1/openapi.json|docs|health` are registered on `app` OUTSIDE the `routes` const (not chained into it) so they are NOT in AppType (§1.6/§4). `apps/api` is compiled to `dist`; `@docket/types/api` re-exports `type AppType` only. Both api and consumer tsconfig set `strict: true` and `moduleResolution: bundler|nodenext` consistently. Hono is pinned byte-identical via the root catalog.

**Why.** §1.2/§1.3 and build-sequence Part 0 are explicit that RPC inference survives ONLY with unbroken method chaining, split files, compiled dist, strict:true on both sides, and a single pinned Hono. §1.6 requires orgContextMiddleware only on /orgs/:orgId/_, and §4 requires auth/mcp/docs to be outside AppType. Applying the middleware via `orgs.use('/:orgId/_', …)` before the child routes is the only placement that both keeps the chain intact and guarantees actorCtx is set for nested handlers.

### §1.7 errors: 'Hono onError maps ZodError→422 … else 500; validator failures use the validator hook to emit the same Problem shape.' Confirm the exact Problem schema, status mapping, and how hono-openapi's validator hook produces fieldErrors.

**Decision.** Canonical `Problem` (RFC 9457) in `@docket/types`: `{ type: string (URI ref, default 'about:blank'), title: string, status: number, detail?: string, code: string (machine-readable, e.g. 'task_already_linked'), fieldErrors?: Record<string, string[]> }`. Status map in `apps/api/src/error.ts` `onError`: `ZodError`→422 (output-validation failures), `AuthError`→401, `CapabilityError`→403, `NotFoundError`→404, `ConflictError`→409, `CycleError`→409 (code `dependency_cycle`), `IdempotencyConflictError`→422 (code `idempotency_key_reuse`), `BillingFrozenError`→402 (lifecycle pending_deletion/past_due write-block, per permissions §0.1), else→500. INPUT validation: pass a `hook` to every `validator(target, schema, hook)` that, on `!result.success`, flattens `result.error` via `z.flattenError` into `fieldErrors` and throws/returns the 422 `Problem` (so the body matches onError's shape exactly). The `Content-Type` on all error responses is `application/problem+json`. `Problem` is attached to every route's 4xx/5xx in `describeRoute` so the OpenAPI doc shows it. Codes are a closed enum exported from `@docket/types` so clients can switch on them.

**Why.** §1.7 specifies the RFC 9457 Problem shape and the exact status mappings; using one shared hook + onError emitting the identical shape satisfies the spec's requirement that validator failures and runtime errors are indistinguishable in shape. The 402 BillingFrozen status is added because permissions §0.1 says billing-state gating runs as a separate 402/403 concern before the authz layer, which the API must surface. z.flattenError is the Zod v4 way to build fieldErrors (the spec pins Zod v4 in §1.4).

### §1.7 soft delete: 'delete routes set archived_at (DELETE method, returns the archived row); hard delete is admin-only.' But several DELETE routes return `{ removed: true }` (grants, roles, members, milestones, dependencies, links, comments, updates, notifications, dailyplan items). Which entities soft-delete vs hard-delete, and what does each DELETE return?

**Decision.** Two classes, fixed: (A) ARCHIVABLE work entities (have `archived_at` via auditColumns): `org` (→pending_deletion lifecycle, special), `team`, `initiative`, `program`, `project`, `task`, `agent`, `integration` — their DELETE sets `archived_at` and returns the archived row shape `{ id, archivedAt }` (org returns `{ id, lifecycleState, deleteAfterAt }`; integration returns `{ id, status:'disconnected' }`). (B) HARD-DELETE join/child/metadata rows that have NO archived_at column (grant, role, member-link, milestone, task_dependency edge, comment, update, notification mark, dailyPlanItem, label, initiative<->project/program links) — their DELETE physically removes the row (or unlinks) and returns `{ id, removed: true }` (or `{ unlinked: true }` for join rows). This is NOT the 'hard delete is admin-only' rule — that rule governs hard-deleting ARCHIVABLE work entities (purging an already-archived task/project), which is exposed ONLY under `/admin` (staff-gated). Comments/updates DELETE is author-or-`org:manage` and is a real row delete (these have no archived_at in data-model §6.7/§6.2) — acceptable because they are immutable-history-light and author-owned; the org `audit_event` retains the trail. Milestone DELETE additionally nulls dependent tasks' `milestone_id` (§3.5).

**Why.** The data-model determines which tables carry archived_at (only auditColumns entities do). Join/child/metadata tables (grant, role, milestone, task_dependency, comment, update, notification, dailyPlanItem, label, m2m links) have no archived_at, so they can only hard-delete — exactly what the `{removed/unlinked:true}` returns indicate. The §1.7 'hard delete is admin-only' refers to purging archived work entities, which §3.16 admin routes own. This reconciles the two return-shape conventions without contradiction.

### §3.1 `POST /orgs` 'card-gate when shared/invited per billing spec' and `members/invite` 'card-gate may apply'. RECONCILIATION defers 'no-card vs card-required trial default' to the billing phase. What is the v1 API behavior so a builder isn't blocked?

**Decision.** v1 API behavior (forward-compatible with the deferred billing decision): `POST /orgs` with `isPersonal:true` or no invites → ALWAYS allowed, no card (personal/solo is no-card per env-and-bootstrap §1.4 note and RECONCILIATION 'Personal space'). `POST /orgs` creating a SHARED org (isPersonal:false) → allowed; the org is created in `lifecycle_state='trialing'` and a Stripe customer is created (referenceId=org.id), but NO card is required AT CREATION for v1 (default = no-card trial). `POST /orgs/:orgId/members/invite` → the FIRST invite on a shared org triggers a soft card-gate: if the org has no active subscription/payment method AND `DOCKET_REQUIRE_CARD_FOR_INVITE` (a boolean env, default `false` in v1) is true → return 402 `Problem` code `card_required` with `{ checkoutHint: { plan, returnUrl } }`; otherwise allow. The gate is thus DATA-DRIVEN by one env flag whose default (no-card) matches the RECONCILIATION default proposal, so flipping the billing decision later is an env change, not a code change. Both routes record the lifecycle state on the org for the billing lane to enforce later.

**Why.** RECONCILIATION defers the no-card/card-required default but the API contract must be unblocking and forward-compatible. env-and-bootstrap §1.4 states personal/solo is no-card and the Team price gates shared orgs/invites. Making the gate an env-flagged 402 with a default of no-card lets the deferred product call be honored by config (dev mirrors prod, env-only deploy — Hard Constraint), with zero code change when billing finalizes.

### §3.12 + §3.12 footnote: integrations POST returns `{ integrationId, authorizeUrl }` and OAuth callbacks live in top-level `integrations/oauth` (public, signed state). What is the exact state/token model, and does it ever reuse the MCP token?

### §3.7 `POST /:taskId/state` and `POST /:taskId/move` overlap with `PATCH /:taskId` (which also accepts state via TaskUpdate? — TaskUpdate has `state?`). What is the canonical way to change state / move, and does PATCH accept state?

**Why.** data-model task has one `state` column and reparenting changes ancestor_path (permissions §7.3) transactionally over the subtree — concentrating reparenting in `/move` prevents PATCH and move from both maintaining ancestor_path (a correctness hazard). Keeping state on both PATCH and /state with shared side-effects matches mcp-surface (update_task patch has no project_id; move_task is separate) and avoids divergence between REST and MCP. All three are RS-side, additive, and behind no feature flag — they ship as the standing MCP integration. Stateful resumable MCP sessions (Redis event store + Fluid Compute) remain deferred to the MCP phase per RECONCILIATION; v1 MCP is stateless request/response + SSE for session streams.

**Why.** Context7 confirms `validAudiences` (RFC 8707) and `verifyAccessToken`/`mcpHandler` token-payload access exist in current Better Auth, so the only real uncertainty is the exact `getMcpSession` helper name/shape and the CIMD metadata flag. Routing ALL token resolution through one `resolveMcpPrincipal` shim makes the RS resilient to the helper's exact signature, and hardcoding the well-known field is trivial since the RS already serves that document. This eliminates the "verify or shim" ambiguity with a plan that works either way.

### mcp-surface §2.2 open issue: "Why global (not org-qualified) scopes… Org is a call argument, authorized at execution. (See open issue.)" — are MCP scopes global or org-qualified?

**Decision.** GLOBAL scopes, definitively. The 4 scopes (`work:read`, `work:write`, `agents:run`, `connectors:link`) are token-level and org-agnostic; org is ALWAYS a per-call argument (the `docket://{slug}/...` URI or the tool's `org` slug arg) re-authorized at execution via `canActor` against the resolved Actor in that org. Two-layer rule is mandatory and fixed: token scope is NECESSARY (coarse capability-class gate) but NEVER sufficient; per-org `canActor` grant resolution ALWAYS follows. Scope→capability mapping (fixed): `work:read`→requires `view`; `work:write`→requires `contribute` (or `assign` for assign-class tools); `agents:run`→requires `assign` + approver-eligibility on the target; `connectors:link`→requires `manage`. A token with `work:write` still gets per-resource 403/`isError` if the Actor lacks the capability on that specific resource. NO org-qualified scopes (`work:write:org_<id>`) in v1 or later — they would bloat consent and force re-consent on every new membership.

**Why.** The architecture narrative and mcp-surface both state the product model is "one global User, org access via Actor membership" and that org-qualified scopes "would bloat the consent screen and require re-consent on every new membership." Global scopes + execution-time org authorization is the only design consistent with the cross-org Hub (one token, many orgs) and the §10 "scope necessary but not sufficient" rule. This pins the open issue to global.

### §7.5 / §7.1 existence-hiding: the middleware double-calls canActor (once for required, once for 'view') to decide 404 vs 403. Is this the canonical pattern, and how to avoid the extra DB round-trip?

**Decision.** Canonical pattern: `canActor` returns `ResolveResult { allow, reason, effectiveCapability: Capability | null }` and the middleware decides 404-vs-403 from the SINGLE call's `effectiveCapability` — NO second DB call. Rule: if `!allow` then return 404 IFF `effectiveCapability === null OR !satisfies(effectiveCapability,'view')` (actor cannot even see it → hide existence), else 403 (actor can see it but lacks the required capability). Because `canActor` computes the full effective capability in one pass (§6), `satisfies(effective,'view')` is a pure in-memory check. The §7.1 code that does a second `canActor(..., 'view', ...)` is REPLACED by reading `res.effectiveCapability`. This applies to point checks (§7.1) and direct-fetch endpoints; list endpoints (§7.2) inherently hide non-permitted rows (they never appear) so no 404/403 question arises. MCP `resources/read` maps both no-grant and not-found to JSON-RPC `-32002` (mcp-surface §3.6 — never leak existence), which is consistent.

**Why.** The spec's own §6 already returns `effectiveCapability` in `ResolveResult`, making the second `canActor` call in §7.1 redundant and a needless DB round-trip. Deriving 404-vs-403 from the single result is correct, cheaper, and removes the only place the spec issued two resolutions per request — eliminating a performance footgun while preserving the existence-hiding guarantee.

### §4.5 invariants 'Last-owner guard' and 'No self-escalation' are stated but not made enforceable. How exactly are they enforced and where?

**Decision.** Enforce both at WRITE time in `@docket/authz` write-guards imported by every member/role/grant mutation endpoint (API + MCP). LAST-OWNER GUARD: any operation that would remove, suspend, archive, or downgrade (role change to non-owner) the LAST `actor{kind:'human', status:'active'}` whose `role.key='owner'` in an org is rejected with `409 code:'last_owner_guard'`. Implemented as a `SELECT count(*) ... FOR UPDATE` of active owner actors inside the same SERIALIZABLE tx as the mutation (count must remain ≥1 post-op). Applies to: `PATCH /members/:actorId` (role/status change), `DELETE /members/:actorId`, and org `transfer_ownership`. NO-SELF-ESCALATION: on any `grant` write or `role` create/update, the server computes the WRITER's own effective capability on the target resource via `canActor(writer, 'manage', target)` AND asserts the granted/role `base_capability` rank ≤ the writer's effective capability rank on that resource (you cannot mint a grant/role more powerful than yourself on that resource). Rejected with `403 code:'self_escalation'`. Both guards live in `packages/authz/src/write-guards.ts` and are unit-tested. The OWNER-ONLY privileged actions (§4.2: `org.delete`, `org.transfer_ownership`, `org.change_billing_subject`, `role.modify_system_role`) are gated by an additional `requireOwner` check (`actor.role.key==='owner'`) layered BEFORE the capability check on those specific endpoints — `manage` alone is insufficient (an Admin holds `manage` but is blocked).

**Why.** The spec states these as invariants but a builder needs the exact rejection codes, transaction isolation, and call sites. SERIALIZABLE + FOR UPDATE on the owner count prevents a TOCTOU race that could otherwise orphan an org (two concurrent downgrades). The self-escalation check reuses the existing `canActor` engine, keeping one source of truth. The Owner-vs-Admin privileged-action gate is already specified (§4.2) and must be a pre-check, not folded into the capability chain since both hold `manage`.

### §6 step 0: cross-org actor ⇒ deny, suspended actor ⇒ deny. How are these wired given the actor.status enum and multi-tenant boundary, and what about the agent's accountable-owner suspension?

**Decision.** Fixed pre-checks in `canActor` before any grant resolution: (1) Load actor by id; if `actor IS NULL` OR `actor.organization_id !== orgOf(target)` → `deny('cross-org-or-unknown-actor')` (this is the hard tenant boundary; an actor id from org A can never resolve a capability on an org-B resource). (2) If `actor.status !== 'active'` (i.e. `'suspended'`) → `deny('actor-suspended')` — applies to BOTH human and agent actors. (3) For AGENT actors additionally: if the agent's `accountable_owner_id` actor is suspended/archived, the agent is NOT auto-denied (the grant is on the agent actor, not the owner) — but the §9 approval gate's `resolveApprovers` skips suspended approvers. The orgContextMiddleware (API) already 404s when no human Actor exists for `(user.id, orgId)`, so cross-org never reaches handlers via REST; `canActor`'s own check is the defense-in-depth + the MCP/Session-executor path where there is no orgContextMiddleware. `loadActor` joins `actor → role` to populate `{kind, orgId, roleId, roleKey, roleBase, roleDefaultVisibility, status}` in one query.

**Why.** The spec's §6 pseudocode names both denies but a builder needs to know they are unconditional pre-checks (not subject to grant override) and that they apply to agents too. The cross-org check is the literal multi-tenant boundary the whole architecture protects ("organization_id comes only from verified context"); making `canActor` itself enforce it (not just the middleware) covers the MCP and Session-executor call sites that bypass orgContextMiddleware.

### §9.3/§9: approve/reject is itself authorized — caller must be in resolveApprovers AND pass canActor(approver,'assign',target). At apply time the agent's canActor is re-checked. What if grants changed and the agent now lacks capability at apply?

**Decision.** Fixed apply-time semantics: on APPROVE, the Session executor re-runs `canActor(agentActor, requiredCapabilityFor(write), write.target)` AT apply time. If it now FAILS (grant revoked/expired between proposal and approval), the executor does NOT apply the write; it transitions the `session_activity.approval_status` to a terminal `rejected` with `body.reason='capability_revoked_at_apply'`, records an `audit_event {type:'rejected', actor_id: agent, initiator_id: approver}`, and surfaces a `needs_grant` follow-up notification to a `manage`-holder. The approver's authorization (`in resolveApprovers AND canActor(approver,'assign',target)`) is checked at the moment of the approve action, not re-checked at apply (the approver already acted). On REJECT → `approval_status='rejected'`, no write, audited. APPROVAL SCOPE: the API's `scope: 'this'|'all_in_session'` means approving `all_in_session` pre-authorizes all CURRENTLY-proposed actions in that session at approve time (each still individually capability-re-checked at its own apply); it does NOT blanket-authorize future not-yet-proposed actions.

**Why.** The spec says "re-checking canActor for the agent at apply time, in case grants changed" but doesn't say what happens on failure. Failing closed (don't apply, mark rejected, trigger grant-on-request) is the only safe choice consistent with deny-by-default and the two-axis independence (capability axis can revoke after approval). Defining `all_in_session` to bind only currently-proposed actions prevents an approval from silently authorizing an agent's unbounded future writes.

### §9.2/§9.3: a routed approver must hold 'assign' on the target or be filtered out, falling through. What if ALL routing layers resolve to actors who lack 'assign' (empty approver set)?

**Decision.** Fixed fallback chain terminates at ORG OWNERS who, by definition, hold `manage` (⊇ `assign`) org-wide, so the approver set is NEVER empty in a well-formed org (last-owner guard guarantees ≥1 active owner). Concrete: `resolveApprovers` filters each layer's candidates by `canActor(candidate,'assign',target).allow`; if a layer yields an empty filtered set, fall to the next layer; the final layer = active org Owners (always non-empty). If — defensively — even that is empty (should be impossible given the last-owner guard), the agent write is BLOCKED with `outcome:'needs_grant'`-style `blocked: no_eligible_approver` and an `audit_event`, never auto-applied. An `act_with_approval` write is therefore guaranteed to either route to a real authorized approver or block — never silently apply.

**Why.** The spec describes the filter-and-fall-through but leaves the terminal empty-set case open. Anchoring the terminal layer on org Owners (guaranteed ≥1 by the last-owner guard, holding manage⊇assign) closes the gap deterministically. Failing closed on the impossible empty case preserves deny-by-default.

### §6: "Open product item (plan §10): how `Program` renames per skin — provide the override slot now; default keeps 'Program'."

**Decision.** `program` is a first-class skinnable key in `VocabularySkin` exactly like every other entity. The slot exists now and is honored by `useVocabulary('program')`. Canonical defaults per preset: Default/startup = {singular:'Program', plural:'Programs'}; Nonprofit = {singular:'Program', plural:'Programs'} (kept — nonprofits use the word natively); Agency = {singular:'Retainer', plural:'Retainers'}. No component hardcodes 'Program' — the sidebar nav item, group headers, create menu, and detail breadcrumb all route through `useVocabulary('program')`. If a skin omits `program`, the resolver falls back to the Default's 'Program'/'Programs'.

**Why.** The spec explicitly says to provide the override slot now and default to 'Program'. Making it a normal key (no special-casing) is the lowest-judgment build: identical mechanism to `initiative`/`project`. Agency='Retainer' is the only non-trivial override and matches agency domain language already implied by the Agency preset; nonprofit keeping 'Program' matches the spec's own example (`program→"Program" (kept)`).

### Vocabulary preset names mismatch across specs: design-system §6 calls them 'Default / Nonprofit / Agency', but the API `OrgCreate` DTO and data-model default use 'startup' / 'nonprofit' / 'agency' (`presetStartup`).

**Decision.** The canonical preset _keys_ are `startup` | `nonprofit` | `agency` (matching the api-rpc-contract `OrgCreate.vocabulary` enum and the data-model `presetStartup` default). 'Default' in the design-system prose is renamed to `startup` (the startup preset IS the default). **Why.** RECONCILIATION is the tie-breaker and the api-rpc-contract + data-model both encode `startup`/`nonprofit`/`agency`. The design-system spec must not contradict the API contract it serves; renaming 'Default'→'startup' (which is the default) resolves the conflict in favor of the wire/DB names with zero behavioral change.

### GAP — `VocabularySkin` is declared as a Zod-derived `$type<VocabularySkin>()` in data-model but its exact shape is never defined; design-system owns the vocabulary skin so it must pin the type.

**Decision.** Define one canonical Zod schema, source-of-truth in `@docket/types` (`vocabulary.ts`), re-derived by `@docket/db` for `$type` and consumed by `@docket/ui` `useVocabulary`. Shape: `VocabularySkin = { preset: 'startup'|'nonprofit'|'agency'; overrides?: Partial<Record<VocabKey, { singular: string; plural: string }>> }`. `VocabKey = 'initiative'|'program'|'project'|'cycle'|'task'|'milestone'|'team'|'triage'|'myWork'|'views'|'agents'|'settings'`. Resolution order in `useVocabulary(key,{count})`: `org.vocabulary.overrides[key]` → `preset[key]` → `presetStartup[key]`; pluralize by `count`. Presets are full `Record<VocabKey,{singular,plural}>` constants. `presetStartup` = the all-default English nouns (Initiative/Program/Project/Cycle/Task/Milestone/Team/Triage/My Work/Views/Agents/Settings).

**Why.** The data-model defers the shape to a 'Zod-derived $type'; design-system is the owner. A `{preset, overrides}` shape is the minimal structure that supports (a) the three presets the API enum already requires and (b) per-org custom relabeling without forcing every org to store all 12 keys. Single source in `@docket/types` keeps DB `$type`, API DTO, and UI resolver on one schema (consistent with RECONCILIATION's 'one branded type / one Zod source' discipline).

### GAP — `HubPreferences` is declared as `$type<HubPreferences>()` (default `{}`) but its shape is undefined; design-system stores theme, density, rail order/pins, sidebar/properties collapse, and per-view collapse state in `Hub.preferences`.

**Decision.** Canonical Zod schema `HubPreferences` (source in `@docket/types`, `$type` in `@docket/db`), all fields optional with documented client defaults so `{}` is valid: `{ theme?: 'light'|'dark'|'system' (default 'system'); density?: 'comfortable'|'compact'|'spacious' (default 'comfortable'); railOrder?: string[] (org/space ids; default = membership order, Personal last); railPins?: string[] (default []); sidebarCollapsed?: boolean (default false); sidebarWidth?: number (200–320, default 240); propertiesCollapsed?: boolean (default false); propertiesWidth?: number (260–420, default 300); landing?: 'today'|'inbox'|'portfolio'|{ orgSlug: string } (default 'today'); viewState?: Record<string, { collapsedGroups?: string[]; columns?: string[]; groupBy?: string; subGroupBy?: string; sortBy?: string }> }`. `viewState` is keyed by a stable `viewKey` (`'<context>:<screen>'`, e.g. `'org:<orgId>:my-work'`).

**Why.** data-model line 142 says preferences 'holds landing config, rail order/pins'; the design-system spec adds density (§1.7 'persisted in Hub.preferences'), theme (§2.3), region collapse/width (§4.1 'persist to Hub.preferences'), and per-view collapse (§5.1 'persisted per view'). All of these are user-global (cross-org) so they belong on `hub`, not `organization`. Every field optional keeps the `.default({})` valid and lets the client supply defaults, avoiding a migration when fields are added.

### §1.6: org accent reads `org.accentOklch` / `Organization.avatar` dominant color, but `organization` has no accent column (only `avatar text`). How is `--org-accent` derived and where stored?

**Decision.** Do NOT add a DB column. Derive the org accent deterministically client-side from the existing `organization.id` (a ULID, always present): `accentIndex = hashToIndex(org.id, ORG_ACCENT_PALETTE.length)` where `ORG_ACCENT_PALETTE` is a fixed 8-entry OKLCH array in `@docket/ui/src/lib/org-accent.ts` (all hues at chroma/lightness tuned to pass the same contrast bar as `--primary`, distinct from health/priority/state hues). `getOrgAccent(org)` returns `org.avatar`-derived dominant color ONLY if/when avatar extraction is later added; for v1 it returns `ORG_ACCENT_PALETTE[accentIndex]`. Personal space and Hub context use `var(--primary)`. The value is written as the inline `style={{'--org-accent': accentOklch}}` on the AppShell root on rebind (exactly as §1.6 shows). Forward-compat: if a stored accent is ever wanted, add nullable `organization.accent text` later and prefer it over the hash — no schema change needed for v1.

**Why.** RECONCILIATION's env-var-only/greenfield posture and 'no stubs' rule argue against a half-built avatar-dominant-color pipeline. A deterministic hash over the ULID is stable across reloads, requires zero new columns or migrations, guarantees the same org always gets the same hue, and is purely decorative (§8: 'Org accent is decorative only') so it carries no accessibility-information load. A fixed curated palette keeps accents on-brand and contrast-safe, unlike arbitrary avatar extraction.

**Decision.** Custom virtualized flattened tree built on `@tanstack/react-virtual` ONLY. Drop `react-arborist` from `@docket/ui` dependencies entirely. The `ListView` flattens (group-header + sub-group-header + row) into a single virtual list; collapsed groups omit their descendants from the flattened set; selection/keyboard via `useListKeyboard`; reorder via `@dnd-kit`.

**Decision.** Color tokens are keyed by the model's `WorkflowState.type` taxonomy, not by the default state _keys_. ### §2.3 + cross-spec: theme application and FOUC posture.

**Decision.** Theme application follows the browser color scheme in CSS via
`@media (prefers-color-scheme: dark)`. Do not use `next-themes`, root `.dark` synchronization,

### GAP — a `PATCH`/`PUT` endpoint to persist `Hub.preferences` (theme, density, rail order, collapse, view state) is referenced by the design system but absent from the api-rpc-contract.

**Decision.** Canonical endpoint: `PATCH /v1/hub/preferences` with `json: HubPreferencesPatch` (a deep-partial of `HubPreferences`), returning `{ preferences: HubPreferences }`. Auth = `authenticated` only (operates on the caller's own `hub`, no org capability). It deep-merges the patch into `hub.preferences` (object/array fields replaced wholesale at the top level; `viewState` merged per-`viewKey`). Reads come via the existing `GET /v1/hub/*` surfaces / a `GET /v1/hub/preferences`.

**Why.** The design system repeatedly says state 'persists to Hub.preferences' but the api-contract mounts `/hub` only for aggregation reads. A single authenticated PATCH on the caller's own hub is the minimal, capability-free surface consistent with the contract's pattern (cross-org Hub routes require 'no per-resource capability beyond authenticated'). Deep-partial keeps every persistence call cheap and avoids clobbering sibling preferences.

### §4.5 / §5.1: per-view collapse state and region collapse persistence location, and reorder-disabled-when-computed-sort behavior — where exactly stored and how keyed.

**Decision.** All UI-state persistence lives in `Hub.preferences` (cross-device), NOT localStorage-only and NOT on `organization`: region collapse/width → `HubPreferences.sidebarCollapsed/Width`, `propertiesCollapsed/Width`; per-view group-collapse + visible columns + group/sort → `HubPreferences.viewState[viewKey]` where `viewKey='<context>:<screen>'` (Hub screens use `'hub:today'` etc.; org screens use `'org:<orgId>:<screen>'`; saved Views use `'view:<viewId>'`). Manual reorder (`@dnd-kit`) is enabled ONLY when the active `sortBy` is a manual/`sort`-column or `'manual'`; for any computed `sortBy` (priority, due, updated, etc.) the drag handles are hidden and reorder is a no-op, with a `Tooltip` explaining 'Switch to manual sort to reorder.'

**Why.** §5.1 says collapse state is 'persisted per view' and §4.1 says region collapse 'persists to Hub.preferences'; both are user-global, so `Hub.preferences.viewState` (defined above) is the single home. A stable composite `viewKey` namespaced by context prevents collisions between same-named screens across orgs. Disabling reorder under computed sort matches the spec's explicit rule and avoids the incoherent state of dragging rows that an active sort would immediately re-order.

### §1.5 / §2.4: `tw-animate-css` vs deprecated `tailwindcss-animate`, and motion tokens — confirm and pin the shadcn-v4-correct animation setup.

**Decision.** Use `tw-animate-css` (imported via `@import "tw-animate-css";` at the top of `globals.css`, right after `@import "tailwindcss";`) — `tailwindcss-animate` is forbidden (deprecated, Tailwind v3-era). ### §2.1 inline note: 'Token aud MUST be bound to the RFC 8707 resource param (verify; open issue).' and §2.5 step 2 fallback: 'If the AS does not stamp aud from resource, fall back to issuer+client binding + explicit resource allowlist.'

**Decision.** Audience binding is REQUIRED and treated as available: the AS is configured with oauthProvider({ validAudiences: [MCP_RESOURCE_URL] }) and clients run PKCE/S256 carrying resource=MCP_RESOURCE_URL; the RS rejects any token whose aud != MCP_RESOURCE_URL with HTTP 401. The §2.5 'issuer+client binding + resource allowlist' fallback is NOT a runtime branch — it is the documented break-glass only if a specific pinned build is proven to drop aud stamping in operational testing; in that case the RS still hard-rejects unless (iss == MCP_ISSUER_URL AND client_id is in the pre-registered first-party allowlist), and the build is treated as a release blocker to fix. No call is ever served on a token lacking a Docket-issued aud or recognized issuer.

**Why.** validAudiences is confirmed in current Better Auth (ctx7, 2026-06-05), and env spec §1.5 defines MCP_RESOURCE_URL = ${API_URL}/mcp as the canonical audience asserted at boot and in tests. Engineering plan §4 (line 446) names aud binding + no-passthrough as the two confused-deputy guards, so a permissive fallback would defeat the spec's core security claim — the fallback must be strictly bounded to pre-registered first-party clients and flagged as a defect, not a steady state.

### §2.2 open issue: whether MCP scopes should be global vs org-qualified (e.g. work:write:org\_<id>).

**Decision.** GLOBAL/flat scopes, fixed at exactly four: work:read, work:write, agents:run, connectors:link. Org is NOT encoded in the scope. Org is a per-call argument (the docket://{slug} addressing for resources; the org slug field for tools), authorized at execution time by resolving the caller's Actor for (user_id, organization_id) and running the canActor grant cascade. No org-qualified scope variants are ever minted, advertised, or accepted.

**Why.** mcp-surface §2.2 'Why global' and permissions §10 ('MCP scope set is a coarse gate layered above canActor') and engineering plan §4 (line 343, 'flat global scopes') and architecture §MCP all converge on flat scopes + per-call org authorization. RECONCILIATION does not override this, so the spec's resolution stands. Two-layer auth (scope necessary, canActor sufficient) is the invariant.

### §5 / §6 step-6 open issue: 'Adopt MCP Tasks behind a feature flag (experimental churn).' — whether to ship Tasks in v1.

**Decision.** Tasks are DISABLED in v1. MCP_TASKS_ENABLED defaults to false in every environment (dev mirrors prod). When false: do NOT advertise the 'tasks' capability object in the initialize response, and tool definitions for trigger_agent_session and run_view set execution.taskSupport: 'forbidden' (not 'optional'). **Why.** RECONCILIATION line 38 and engineering plan §4 (line 201, 'Tasks deferred behind MCP_TASKS_ENABLED') and architecture line 316 all defer Tasks. The spec already gates it behind the flag; this pins the default OFF and makes the capability advertisement + taskSupport consistent so there is no half-on state. 'optional' taskSupport while the capability is unadvertised would be an inconsistent surface; flipping both together avoids it.

### §4.1 / spec body: resource URI scheme — custom non-fetchable docket:// vs fetchable https://.

**Decision.** Custom non-fetchable docket://{org}/{type}/{id} scheme, RFC 3986 compliant. Clients MUST read via resources/read (never direct HTTP GET on the URI). {org} is the Organization slug; Personal space = 'personal'; Hub-scoped reads use the literal 'hub' (docket://hub/today, docket://hub/inbox, docket://hub/portfolio). Mutation tools additionally return a human https://app.docket.app/... deep link in EntityRef.url for navigation, but that URL is never an MCP-readable resource. The embedded slug is for addressing only and is NEVER trusted for access — the RS re-authorizes from the token's sub.

**Why.** mcp-surface §4.1 resolves to the non-fetchable scheme; architecture §MCP (line 182) and engineering plan §4 (line 201) confirm 'resolved to the non-fetchable custom scheme.' RECONCILIATION line 7/16 confirm slug-based MCP addressing alongside id-based REST. Settled across all sources.

### Implicit ambiguity: mcp-surface §3.3 and several tool schemas write Id = z.string().uuid(); RECONCILIATION line 7/13/16 mandates ULID text ids repo-wide.

**Decision.** MCP tool/resource schemas MUST import the single branded Id primitive from @docket/types (ULID-shaped, 26-char) — NOT z.string().uuid(). Every occurrence of z.string().uuid() in mcp-surface.md (the Id fragment in §3.3 and all inlined Id uses) is overridden to the shared branded Id. idempotency_key (Idem) is likewise the branded Id type (or a ULID-shaped string), not a UUID. **Why.** RECONCILIATION line 7 (text ULIDs repo-wide, no Postgres uuid) and line 13 (one branded Id in @docket/types consumed by BOTH API and MCP; 'MCP must NOT use z.string().uuid()') are tie-breaker decisions. Architecture line 182/216 restate it verbatim ('MCP tool schemas must not assert z.string().uuid()'). The spec's draft UUID schemas are explicitly flagged as overridden draft text.

### §2.4 SSRF guard + domain trust policy for CIMD: needs a concrete allowlist/rejection rule rather than 'apply an SSRF guard + domain trust policy.'

**Decision.** CIMD resolution enforces, in order, before honoring any URL-form client_id: (1) Scheme MUST be https. (2) Host MUST resolve to a public IP — reject if it resolves to private (RFC1918), loopback (127.0.0.0/8, ::1), link-local (169.254.0.0/16, fe80::/10), unique-local (fc00::/7), or any non-global address; re-check after DNS resolution to defeat rebinding, and pin/verify the resolved IP for the actual fetch. (3) client_id string MUST equal the fetched document's URL byte-for-byte. (4) The host MUST match the MCP_CIMD_TRUST_ALLOWLIST (env var: comma-separated host suffixes; in dev/prod it contains Docket-known client domains, e.g. claude.ai, plus any first-party agent host). Unknown but otherwise-valid public CIMD documents are accepted only with a registered, throttled fetch (max 1 fetch/client_id/5min, 5s timeout, 64KB cap, redirects disabled) AND still require passing the allowlist when MCP_CIMD_STRICT=true (default true in prod). redirect_uris in the document are validated against the same https+public-host rules. Failures return OAuth invalid_client at the AS.

**Why.** mcp-surface §2.4 and engineering plan §4 (line 201) mandate 'SSRF guard + domain-trust allowlist' but leave the rule abstract; this makes it buildable with zero judgment. The new env vars (MCP_CIMD_TRUST_ALLOWLIST, MCP_CIMD_STRICT) follow env spec §1.5's pattern (composed from §0 domains, validated at boot, dev mirrors prod) and are additive to the existing MCP_ALLOWED_ORIGINS guard. Defaulting strict in prod matches the spec's 'reject private/link-local hosts.'

### §3.1 / §3.2: which tool annotations are authoritative given the spec lists both a quick-reference table (§3.2) and per-tool annotation lines (§3.4) that occasionally read tersely (e.g. move_task '...readOnly:F').

**Decision.** The §3.2 annotation quick-reference TABLE is authoritative for all 26 tools; the per-tool §3.4 lines are illustrative and where terse must be expanded to set ALL FOUR ToolAnnotations explicitly (readOnlyHint, destructiveHint, idempotentHint, openWorldHint) — never relying on SDK defaults. Canonical per the table: only link_external, start_connector_link, and trigger_agent_session set openWorldHint:true; everything else openWorldHint:false. destructiveHint:true only on remove_task_dependency, approve_action, reject_action, cancel_session. readOnlyHint:true only on run_view and search (which then omit destructiveHint as N/A but still set the field explicitly to false). Every tool also sets a human-readable title.

**Why.** mcp-surface §3.1 mandates 'We set all four explicitly to avoid relying on defaults' and architecture §MCP (line 182) restates it ('every tool sets all four ToolAnnotations explicitly... destructiveHint defaults to true'). The table is the complete enumeration; the prose lines are partial. Choosing the table as authority removes the ambiguity from terse lines.

### §3.1 error model: when does a failure become a JSON-RPC protocol error vs an isError:true tool result vs an HTTP 403?

**Decision.** Three-tier, fixed mapping: (1) Transport/token failures (missing/invalid bearer, aud mismatch, wrong issuer, missing required SCOPE for the tool) -> HTTP status at the transport (401 for auth, 403 for insufficient_scope) with the §2.6 WWW-Authenticate challenge — these never reach the tool layer. (2) Per-resource GRANT failures (token scope OK but canActor denies on the target row), input/validation failures, and business-rule failures (e.g. dependency cycle, invalid state key, not-found-but-authorized) -> tool result with isError:true + actionable text, so the model can self-correct or request access. (3) Unknown tool name / malformed JSON-RPC / protocol violation -> JSON-RPC protocol error. For resources/read specifically: not-found AND no-grant both return JSON-RPC -32002 with identical not-found semantics (never reveal existence to an unauthorized caller).

**Why.** mcp-surface §3.1 and §4.3 specify exactly this split (scope failure = HTTP/transport, grant failure = isError so the model self-corrects, read no-grant = -32002 not-found to avoid existence leakage). Permissions §10 confirms scope is a coarse gate above canActor and architecture line 197 confirms the not-found-on-no-grant rule. This codifies the spec's prose into one unambiguous decision table.

### §4.4 subscriptions: which resources are subscribable and how change events are sourced under the no-Redis v1 decision.

**Decision.** Subscribable resources in v1: docket://{org}/session/{id}, docket://{org}/task/{id}, docket://hub/inbox, and docket://hub/today. On change the RS emits notifications/resources/updated { uri } over the subscribing connection's GET-SSE channel. Change events are sourced from the SAME service-layer entity-change event bus that drives the web app's realtime and the per-org session SSE stream — NOT from a separate Redis pub/sub. Because v1 transport is stateless/non-resumable, subscriptions are bound to a live MCP connection; if the connection drops, the client re-subscribes and re-reads (no missed-event replay guarantee in v1 — acceptable because resources/read always returns current truth). notifications/resources/list_changed fires (debounced) when the principal's visible-entity set changes materially. resources.subscribe:true and resources.listChanged:true are advertised.

**Why.** mcp-surface §4.4 names these subscribables and the service-layer event-bus fan-out; RECONCILIATION line 29 ('one mechanism... SSE for session streaming, no Redis') forces the event source to be the existing in-process/Postgres-NOTIFY-or-equivalent service bus, not Redis. Binding subscriptions to the live connection is the necessary consequence of the stateless transport decision, and re-read-on-reconnect is sound because docket:// resources are non-fetchable JSON projections of current DB state.

### §5 capabilities: prompts capability — confirm it is intentionally omitted in v1.

**Decision.** Do NOT advertise the 'prompts' capability in v1. No server-defined prompt templates are registered. This is deliberate and final for v1; implementers must not add it speculatively. The advertised capability set is exactly: tools{listChanged:true}, resources{subscribe:true,listChanged:true}, completions{}, logging{}, and tasks{...} ONLY when MCP_TASKS_ENABLED=true (otherwise omitted entirely).

**Why.** mcp-surface §5 explicitly states prompts is NOT advertised in v1 and lists it 'so implementers don't add it speculatively.' Combined with the Tasks-default-off decision above, this fully pins the initialize capabilities object with no remaining ambiguity.

### §6 step-8 / env spec §1.5: reconcile the env-var contract for MCP — mcp-surface lists MCP_CANONICAL_URL and MCP_SESSION_STORE_URL, while env spec §1.5 uses MCP_ISSUER_URL, MCP_RESOURCE_URL, MCP_ALLOWED_ORIGINS, OIDC_LOGIN_PAGE_URL.

### §2.4/§2.7 the spec's passkey-first flow calls `authClient.passkey.addPasskey({ name, context })` and `signIn.passkey()`, but verification (ctx7, Better Auth 1.6.14 docs) shows that with `registration.requireSession:false` the `context` is a REQUIRED server-validated signed token and `generatePasskeyRegistrationOptions({ context })` must mint it. HMAC-signing with the existing `BETTER_AUTH_SECRET` needs no new infra and ties identity birth to a server-issued intent. Transactional user+hub creation is mandated by engineering plan §2 ('resolveUser must create both transactionally').

### §2.2 task default state: the spec says 'state default first workflow state' and notes the new backlog/todo/in_progress/done/canceled model, while RECONCILIATION #10 says 'app sets state = team.workflow_states[0].key; the default team's first state is backlog. No DB default column.' and data-model embeds workflow states as typed jsonb. The slice handler behavior must be exact.

**Decision.** `task.state` has NO DB default column. On `POST /v1/orgs/:orgId/tasks`, if `TaskCreate.state` is omitted, the handler loads the task's team, reads `team.workflow_states` (typed jsonb `WorkflowState[]` = `{key,name,type,position}` ordered by position), and sets `state = workflow_states[0].key`. The default team is seeded with the data-model `defaultWorkflowStates` whose first entry has `key='backlog'` (type `backlog`), so the slice's first task lands in `backlog`. Any explicitly-passed `state` is validated against the set of `team.workflow_states[].key` (api-rpc `POST /:taskId/state` uses the same validation) — an unknown key → 422. List grouping is by Project → by team workflow state, and a task with no `project_id` renders under a 'No project / Triage' group (proving the grouping branch in the slice's step 5).

**Why.** RECONCILIATION #10 is explicit (app-set, no DB default, backlog first) and data-model §3.5 embeds the states as jsonb keyed by position. Reading `workflow_states[0].key` rather than hardcoding 'backlog' keeps the rule correct even for teams with customized state sets, which the Phase-6 Triage/Cycles lanes depend on.

### §2.6 / env-and-bootstrap: the slice's bootstrap (§2.6) lists 5 steps against the full env-and-bootstrap spec which adds DATABASE_URL_UNPOOLED, per-app .env writing, neonctl/stripe/vercel orchestration, and idempotency. The minimal slice bootstrap must be reconciled with the full spec so the same script serves both.

**Decision.** The bootstrap is ONE script (`scripts/bootstrap.ts`, the full env-and-bootstrap §3 flow); the slice runs a SUBSET of its steps, not a different script. For the slice, `pnpm bootstrap` (dev target) must: (1) preflight `neonctl`; (2) provision/reuse a Neon `docket` project + `dev` branch and capture BOTH `DATABASE_URL` (pooled, runtime) and `DATABASE_URL_UNPOOLED` (direct, for migrations) — migrations MUST run over the unpooled string, never the pooler; (3) generate `BETTER_AUTH_SECRET` (`openssl rand -base64 32`) and set `BETTER_AUTH_URL=API_URL`, `BETTER_AUTH_PASSKEY_RP_ID=localhost`, `BETTER_AUTH_PASSKEY_RP_NAME=Docket`, `API_URL=http://localhost:8787`, `NEXT_PUBLIC_API_URL` (mirror), `NEXT_PUBLIC_WEB_URL=http://localhost:3000`; (4) run `pnpm --filter @docket/db db:generate` (Better Auth CLI 1.6.14 schema → `@docket/db`) then `db:migrate` over the unpooled string; (5) validate via `@docket/env` (`pnpm env:check`) and fail loudly on any missing required var; (6) write per-app `.env` atomically (`apps/api/.env` server slice, `apps/web/.env` the NEXT*PUBLIC*\* slice) and emit redacted `.env.example`. Stripe/OAuth/MCP/Athena prompts are SKIPPED for the slice (their Phase-6 lanes enable them); `agentEnv`/`stripeServerEnv` mark those optional for the first run so `env:check` passes with a warning, never an error.

**Why.** env-and-bootstrap §3 is the authoritative, idempotent flow; the slice must not fork it (that would violate dev-mirrors-prod / one-path). The spec's §2.6 omitted `DATABASE_URL_UNPOOLED`, which env-spec §1.1 marks REQUIRED for `drizzle-kit migrate` (migrations over a PgBouncer pooler break DDL) — including it is load-bearing. Marking the not-yet-needed slices optional lets the identical validator pass on a minimal first run while still enforcing the full contract once lanes turn on.
