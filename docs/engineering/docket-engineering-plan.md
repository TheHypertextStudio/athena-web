# Docket — Engineering Plan

> **Companion to** `docs/core/mvp-plan.md` (the product plan). This document is the _how_: stack, architecture, auth/identity, billing, the MCP server, the data model, and infrastructure. Grounded against current documentation (workflow research, 2026‑06‑04/05); pin versions and re‑verify before upgrading.

---

## 0. Hard Constraints

These four constraints are upstream of every other decision in this plan. They are not preferences to be relitigated per task; they are the boundary conditions the rest of the document assumes. Each one exists to rule something _out_ — a tempting shortcut, a deprecated API, a dev-only code path, a quality gap — so stating them explicitly is what keeps the build from drifting back toward those defaults under time pressure. (The structural consequences — the single Hono backend, the same-origin rewrite, the `env → db → auth → api` build spine — are narrated in architecture.md §"One backend"; this section is the _why_ behind the constraints those structures satisfy.)

### 1. Greenfield

We are **not** attached to the existing `athena-web` codebase. The first build action is to scaffold a fresh **Turborepo** monorepo; the existing code is reference material at most, never a foundation to extend. The reason is that Docket's shape is determined by decisions that touch the lowest layers — a repo-wide `text` ULID id primitive wired into Better Auth's `advanced.database.generateId`, a single `@docket/db` package owning _all_ SQL including the Better-Auth-generated tables, a compiled-versus-JIT package split tuned for Hono RPC type inference. Retrofitting those into an inherited tree means fighting whatever id strategy, schema ownership, and tsconfig posture already exist there, and every such fight is a place where the FK graph or the `AppType` contract silently degrades. Starting clean is cheaper and safer than reconciling. This is why `scripts/bootstrap.ts` (`pnpm bootstrap`) provisions real Neon, Stripe, Vercel, and OAuth resources on a fresh checkout rather than assuming any pre-existing state.

### 2. No legacy code or APIs; upgrade as needed

Everything runs on the latest stable release using modern APIs; no deprecated path is acceptable even when it still technically works. The motivation is that several of these "deprecated but functional" paths are exactly the ones that are removed in the very next major version or that are structurally incompatible with the architecture we have chosen, so adopting them buys a migration we would immediately have to pay back. Concretely:

- **Turborepo 2.x `tasks` key**, never the removed 1.x `pipeline` key. The `pipeline` key does not merely warn — it is gone in 2.x, so a config written against it is a non-starter, not a deprecation. We standardize on **turbo 2.9.x** (see §1) and declare `env`/`globalEnv` so the build cache invalidates on value changes.
- **MCP Streamable HTTP** transport, never the deprecated two-endpoint HTTP+SSE transport. We target spec **2025-11-25**, including **CIMD** (client-id-metadata-documents) as the primary client-registration path, with Dynamic Client Registration `/register` kept only as a **MAY** fallback — DCR was downgraded to MAY in this spec revision, so leaning on it as the primary path would be building on a discouraged mechanism. (The transport's MUST-level guards and the audience-binding rules are detailed in §4 and architecture.md §"MCP".)
- **Playwright ≥ 1.60**, **Next.js 16** + **React 19** with the React Compiler, **Better Auth 1.6.14** (the latest stable line; the 1.7 beta is on the watchlist but there is _no published 2.0 line yet_, so 1.6.14 is the ceiling, not a conservative floor), and **Drizzle latest**. Pinning to current-stable is what lets us write against the modern API surface (App Router conventions, the `passkey({ requireSession:false, resolveUser })` first-onboarding hook, the 2.x task graph) without carrying compatibility shims for paths that are on their way out.

There is one important **caveat that is explicitly not a violation** of this constraint. Stripe's `trial_period_days` — surfaced through the Better Auth Stripe plugin as `plan.freeTrial.days` — is _named_ "legacy trial," but it is the **supported, non-deprecated** mechanism and the **only** one compatible with Stripe Checkout and the `@better-auth/stripe` plugin. The newer **Trial Offer API** is still in preview and is **Checkout-incompatible**, so adopting it would break the embedded-Checkout flow Docket depends on (see §3). We therefore keep the supported trial path (`freeTrial.days: 14`) deliberately and on purpose; the word "legacy" in Stripe's naming is misleading and does not make this a deprecated API.

### 3. Env-var-only deploy (12-factor), dev mirrors prod

When deployed, the service must run from **environment variables alone** — no baked-in config, no machine-specific files, nothing that exists only in one operator's setup. The corollary, and the part that actually does the work, is that **dev mirrors prod**: the same env _contract_ and the same validation run in both environments, and only the values differ (a dev Neon branch, `sk_test_` keys). The point of this constraint is to eliminate the entire class of "passed locally, broke on deploy" failures by refusing to have a separate dev code path at all. Every variable is validated through `@docket/env` (built on the `@t3-oss/env` extends pattern, §1), so importing the env package with a required variable missing throws immediately rather than failing deep inside a request in production. The same single Hono `apps/api` service that runs as a Vercel deployment is the one `pnpm dev` brings up, and the same-origin rewrite that keeps Better Auth cookies first-party is exercised in development too — the failure modes are caught on the developer's machine because the developer's machine _is_ the production topology with different secrets. (architecture.md §"Environment" walks through the per-app `.env` placement and turbo strict env mode that enforce this.)

### 4. Standards & quality

Four non-negotiable quality gates underpin everything: **strict TypeScript** (no implicit `any`; strictness is also load-bearing for Hono RPC inference, which collapses to `any` if any consumer's tsconfig drops `"strict": true`), **Zod validation in _and_ out** (request _and_ response shapes validated against the schemas exported from `@docket/types`, so the same definitions bind the RPC handlers, the MCP tools, and the Next server actions), **OpenAPI/Scalar** for the API surface, **≥ 80% coverage** targets, and an **accessible UI**. These are stated as a hard constraint rather than a "nice to have" because each one is cheap to maintain continuously and expensive to retrofit: a response schema added after the fact rarely matches what handlers already emit, coverage clawed back late tends to test what the code does rather than what it should do, and accessibility bolted on at the end means re-deriving component semantics. Holding the line from the first commit is what keeps the type contract, the API documentation, and the test suite trustworthy as the surface grows.

## 1. Monorepo & Stack (Turborepo)

Docket is built as a single Turborepo on pnpm workspaces with the internal package namespace `@docket/*`, modeled on the `create-t3-turbo` reference but adapted from tRPC to **Hono RPC**. The structural diagram — the full `apps/* + packages/* + tooling/*` tree and the dependency/type-flow graph — lives in `architecture.md` (§ Monorepo Structure); this section is the engineering-decision companion: _why_ each choice is forced, how it is configured, what breaks if you do it differently, and the order in which it must be built. Cross-reference the diagram there rather than expecting it redrawn here.

**Why pnpm + turbo, specifically.** The product is not one app but a federation that must share one database schema, one auth instance, one set of Zod contracts, and one design system without drifting. pnpm's strict, non-hoisted, content-addressed `node_modules` is load-bearing for that: a package can only import what it explicitly declares, which is precisely what lets us guarantee a single, identical Hono version across `apps/api` and every consumer — a hard prerequisite for RPC type inference (below) that a hoisting installer could silently violate by deduping two compatible-but-different Hono versions into one. **turbo 2.9.x** is the task runner and cache on top. The version floor matters: `turbo.json` must use the 2.x **`tasks`** key, never the removed 1.x `pipeline` key (Hard Constraint 0.2) — a `pipeline`-keyed config is not a soft deprecation, it simply will not validate against the 2.x schema. We declare `env`/`globalEnv` explicitly so the build cache invalidates when a relevant environment value changes instead of serving a stale artifact, run in **strict env mode** so a task can only see the variables it has declared, and keep `.env` files **per-app, not at the repo root**, so each deployable's environment surface is isolated and a cache hit can never leak a sibling app's variables.

**The layout, and its deployment consequence.** Four deployables live in `apps/` — `web` (Next.js 16, the product app), `marketing` (Next.js 16, the Linear-grade landing → sign-up surface), `admin` (Next.js 16, a _separate_ service-operator back-office), and `api` (Hono 4.x, the work-layer API plus the `/api/auth/*` mount and the MCP server). Six shared packages live in `packages/`: `db` (Drizzle/Postgres), `auth` (the Better Auth `betterAuth()` instance and framework handlers), `ui` (shared shadcn/Tailwind components), `types` (Zod schemas plus the Hono RPC `AppType` contract), `env` (`@t3-oss/env` validation), and `test-utils`. The `tooling/` directory holds shared `tsconfig`, `eslint-config`, and `tailwind-config` presets — no runtime code, purely strictness/style parity across the workspace. The reason `apps/api` is a standalone Hono service rather than Next route handlers is deliberate: it gets an independent deploy cadence, a clean 12-factor env-var-only deploy, and it is the natural home for the MCP server and the OAuth provider. Its tradeoff — losing automatic same-origin co-location with the Next apps — is bought back at the platform layer via Vercel rewrites (see architecture.md), not paid in CORS complexity by default.

**`@docket/db` as the single SQL owner.** This is the repo's non-negotiable rule: every table — the work-domain tables (`organization`, `actor`, `team`, `project`, `task`, …) _and_ the Better Auth tables (`user`, `session`, `account`, `verification`, `passkey`) — is defined, migrated, and exported from this one package. Better Auth does not get a schema island: `npx @better-auth/cli generate` emits its Drizzle schema _into_ `@docket/db`, and `drizzle-kit migrate` applies it from there (see §2 for the auth-side mechanics). The payoff is enforceability of the ULID decision from RECONCILIATION: every PK is `text("id").primaryKey().$defaultFn(genId)` driven by one generator in `@docket/db/src/id.ts`, with Better Auth's `advanced.database.generateId` wired to that _same_ generator so `user.id` is `text` and lines up with every foreign key — including the `actor.user_id` link that folds membership into the human Actor. If auth owned its own tables, `user.id` would default to Better Auth's own id scheme and the FK graph would fracture; centralizing SQL is the precondition for one id primitive across the whole graph and for `ON DELETE CASCADE` to flow cleanly from `organization` through every `organization_id` column during a lifecycle purge.

**The compiled-vs-JIT split is mandatory, and the reason is `tsserver`.** The packages divide into two compilation regimes:

| Regime                                                     | Members                                      | How consumed                            |
| ---------------------------------------------------------- | -------------------------------------------- | --------------------------------------- |
| **Compiled** (`tsc → dist`, `main`/`types` → build output) | `@docket/db`, `@docket/auth`, `apps/api`     | consumers import a finished `.d.ts`     |
| **JIT** (raw TypeScript, no build step)                    | `@docket/ui`, `@docket/types`, `@docket/env` | consumed via Next's `transpilePackages` |

The compiled side exists _because of_ the Hono RPC contract, not as a packaging preference. RPC type inference only survives if the router is **method-chained** (`app.route(...).route(...)`) and exported as `export type AppType = typeof routes`, and that inferred type is enormous. The failure mode this prevents is concrete: if `apps/web` imported `apps/api`'s router _source_ directly, `tsserver` would re-infer the entire chained-router type on every keystroke and grind to a halt. The defense is to split the router across multiple files (`routes/organizations.ts`, `routes/projects.ts`, `routes/tasks.ts`, and friends) _and_ compile `apps/api` to a flattened `.d.ts`, so the editor reads a pre-computed type instead of recomputing it live. That is the entire justification for the compiled/JIT line — an editor-performance decision. The corollary is that the **Hono version must be identical** across `api` and every consumer; a version skew here desynchronizes the inferred and consumed types and the RPC client silently degrades.

**`AppType` is a type-only flow.** What actually crosses from `apps/api` to the three Next apps is _only_ the `AppType` — a type-only import, never the Hono runtime. The Next apps construct an `hc<AppType>(NEXT_PUBLIC_API_URL, { init: { credentials: 'include' } })` client and get fully-typed `client.organizations.$post`, `client.projects.$post`, `client.tasks.$post` calls with request and response shapes inferred end to end. The non-obvious requirement: **both** the API tsconfig and _every_ consumer tsconfig must set `"strict": true`, or the inference silently collapses to `any` rather than erroring — a quiet failure worth a CI guard. `@docket/types` is the seam underneath, exporting the Zod schemas that `apps/api` validates _both_ request and response against (so the RPC client, the handlers, and the UI reference one definition) plus the single branded `Id` primitive shared by the REST routes and the MCP tool schemas. Runtime requests, by contrast, travel **same-origin** over Vercel rewrites (`apps/web` rewrites `/api/*` → `apps/api`) to keep the Better Auth session cookie first-party; CORS-with-credentials is the fallback only if the deployables must be split (RECONCILIATION).

**`@docket/env` and the extends pattern.** Environment configuration is centralized in `@docket/env`, imported by every package and app, built on `@t3-oss/env-core`. It defines one shared **base contract** — server vars `DATABASE_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, the OAuth client id/secret pairs (`GOOGLE_*`, `GITHUB_*`, `LINEAR_*`), and the client var `NEXT_PUBLIC_API_URL` — that each deployable **extends** so it inherits exactly the variables it needs and no more. Validation runs at boot: importing `@docket/env` with a required variable missing throws _immediately_ in dev rather than failing deep inside a request, which is how the 12-factor contract is enforced rather than merely advised. This composes with the turbo configuration above — strict env mode plus the `env`/`globalEnv` declarations keep the build cache correct when values change, and the per-app `.env` placement means each `extends`-composed deployable sees only its own slice.

**What makes dev mirror prod.** The thread tying the structure together is Hard Constraint 3: the same `@docket/env` schema and the same validation run in both environments — only the values differ. The same `apps/api` Hono service that runs as a Vercel deployment is the one `pnpm dev` brings up locally; the same Neon-Postgres-targeted `@docket/db` migrations apply against a local or Neon database; the same same-origin rewrite path that keeps cookies first-party in production is exercised in development. There is deliberately no separate "dev mode" code path that could pass locally and break on deploy, which is exactly why `scripts/bootstrap.ts` (`pnpm bootstrap`) provisions _real_ Neon, _real_ auth secrets, and _real_ migrations on a fresh checkout instead of stubbing them.

**Build sequencing.** Per `build-sequence-and-slice.md`, the structure above must be assembled along a strictly-sequential spine — `Repo skeleton → @docket/env → @docket/db → @docket/auth → apps/api (RPC + auth mount) → permissions core → @docket/ui shell` — because each link consumes the previous link's compiled output and reordering breaks either type inference or migrations. Phase 0 establishes the skeleton and wires the compilation policy _immediately_ (empty `package.json`s with `@docket/*` names and the compiled/JIT `main`/`types` targets already set), so the dependency graph is declarable and the `tsserver` discipline is in place before any router exists to strain it. Once the three seams — DB schema, the `AppType` + `@docket/types` contract, and the `@docket/ui` shell — are frozen, feature work (Programs, Initiatives, Cycles, Triage, Agents/Sessions, Portfolio, Billing, MCP) fans out in parallel, each lane adding new tables, new chained routes, and new components without re-cutting a seam. Work that _is_ parallelizable earlier — the `tooling/*` presets (alongside Phase 0–1), `@docket/test-utils` and the Playwright harness (alongside Phase 2–4), and the `apps/marketing` static landing (alongside Phase 5, since it only needs `@docket/ui`) — can proceed off the critical path.

**Open decisions:** whether the marketing app consumes `@docket/ui` or stays lean and standalone; the Drizzle pre-1.0 pin and the v1 migration plan.

## 2. Auth & Identity (Better Auth)

Authentication is a single `betterAuth()` instance backed by Postgres through the Drizzle adapter, living in `@docket/auth` and mounted by exactly one service. We pin **better-auth 1.6.14** exactly — not a range — because the surface we depend on is broad (eight plugins, several from separate packages) and because the `mcp()` plugin is slated to migrate into a future "OAuth Provider Plugin"; pinning means an upgrade is a deliberate, tested step rather than a silent dependency bump that could shift the OAuth/MCP framing under us. Watch the 1.7 beta, but treat it as a future migration, not a target (there is no published 2.0 line yet, per §0.2).

### The global-User principle

The identity model inverts the assumption most B2B auth makes. The **User is a global account that persists even after losing every organization membership** — GitHub-style. Deleting a user from one org (or SCIM-deprovisioning them out of all of them) does not destroy the account; it removes _memberships_. The product distinction is **identity ≠ membership**: a person can sign in with zero orgs and still own their personal Hub. This principle drives a structural decision documented in architecture.md §"Domain Model" / RECONCILIATION: the Better Auth **organization plugin is turned OFF**. Docket owns its own `organization` and `actor` tables, where a human `Actor` row folds in membership (`user_id` + `role_id`) with a partial unique index on `(organization_id, user_id)` — so "joining an org" is literally "an `actor` row exists," and the global `User` survives at zero memberships. Invitations are likewise a hand-built `invitation` table, not the plugin's. Better Auth therefore owns only the _global identity_ tables (`user`, `session`, `account`, `verification`, `passkey`); the tenant/membership graph is Docket's. See architecture.md for why this folding is the load-bearing modeling call (it is what lets `actor.user_id` FK cleanly to the global `user.id`, which is why the ULID generator must also drive Better Auth's `advanced.database.generateId`).

### The plugin set and how it composes

All capabilities are configured as plugins in the **one** `betterAuth()` call. Order matters in one specific way: **`nextCookies()` must be last** in the plugin array, because it hooks cookie-setting on the response and must wrap every other plugin's auth responses. The rest of the set divides by _what need it serves_; the table below is the at-a-glance map, and the prose after it explains the composition seams that the table can't carry.

| Need                                    | Mechanism                                                                                                                                                                                  |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Passkeys (primary)**                  | `passkey({ registration: { requireSession: false, resolveUser } })` → passkey-FIRST onboarding (register _before_ a session exists).                                                       |
| **Google + GitHub**                     | Core `socialProviders` + `account.accountLinking.trustedProviders` (these are _links_ to one account, not separate accounts). Keep `trustedProviders` minimal — it is a security boundary. |
| **Linear**                              | Native `socialProviders.linear` entry (Better Auth 1.6.14 built-in; verify the live authorize/token/userinfo URLs + scopes).                                                               |
| **Enterprise SSO**                      | `sso()` — ships both **OIDC and SAML 2.0** (Docket as Service Provider).                                                                                                                   |
| **SCIM provisioning**                   | `scim()` — SCIM 2.0 **Users only (NO Groups)**. If enterprise group→Team sync is needed, map IdP groups via SAML attributes at login instead.                                              |
| **Docket as OAuth 2.1 / OIDC provider** | `oidcProvider()` — PKCE, tables `oauthApplication`/`oauthAccessToken`/`oauthConsent`, RFC 7591 `/register`.                                                                                |
| **Remote MCP server**                   | `mcp()` — built on the OIDC provider; `withMcpAuth`, `getMcpSession`, `oAuthDiscoveryMetadata`/`oAuthProtectedResourceMetadata`.                                                           |
| **Billing**                             | `stripe()` (see §3).                                                                                                                                                                       |

**Passkey is primary, and "primary" means the onboarding sequence is inverted.** `passkey({ registration: { requireSession: false, resolveUser } })` allows a credential to be registered _before_ any session exists — which is exactly what "sign up with a passkey, no password" requires. The `resolveUser` callback is the seam where global identity is _born_: on first passkey registration with no session it creates the `user` row and its 1:1 `hub` row in one step (the per-User personal command center from §5). Everything else in onboarding hangs off that. The failure mode to design against is a partially-created identity (a `user` with no `hub`), so `resolveUser` must create both transactionally. The build sequence treats this as Phase 3's exit gate: a unit test that registers and verifies a passkey via a virtual authenticator and receives a session cookie — and `passkey` + `nextCookies()` are the _only_ plugins needed to prove the first vertical slice; the rest are added in their own Phase-6 lanes.

**Social Google/GitHub and account linking are about one account, not several.** Google and GitHub come from core `socialProviders`; the linking behavior comes from `account.accountLinking.trustedProviders`. The distinction that the table compresses: a trusted provider means "if someone signs in with Google using an email that already has an account, _link_ it to that account" rather than minting a second one. That auto-link is convenient and dangerous in equal measure — it is a security boundary, because a provider you trust to link is a provider that can take over an existing account by email match. Keep `trustedProviders` minimal and only include providers whose email verification you trust. Google here doubles as the sign-in path _and_ the data-source link for Workspace/Drive/Gmail/Calendar connectors; GitHub doubles as sign-in _and_ the GitHub connector — but those connector tokens are issued and stored separately from the login flow (see §4's MUST: never reuse one token across purposes).

**Linear uses Better Auth's native built-in social provider** with provider id `linear`. The provider constants in `@docket/auth` are `https://linear.app/oauth/authorize`, `https://api.linear.app/oauth/token`, and `https://api.linear.app/graphql` (identity comes from Linear's `viewer` query). The account-linking route is Better Auth's native `/api/auth/link-social`, and the callback is `${BETTER_AUTH_URL}/api/auth/callback/linear`. Login requests only `["read"]`; the migration connector may request broader connector scopes separately. The client id/secret are env (`LINEAR_CLIENT_ID`/`LINEAR_CLIENT_SECRET`).

**Enterprise SSO and SCIM are a pair with a deliberate gap.** `sso()` makes Docket a SAML 2.0 / OIDC **Service Provider** (the enterprise IdP authenticates the user into Docket). `scim()` gives the IdP a SCIM 2.0 endpoint to provision and _deprovision_ users — and this is precisely the operation that exercises the global-User principle above: a SCIM deprovision removes memberships, not the account. The pinned limitation: `scim()` is **Users only — no Groups**. There is no automatic IdP-group → Docket-Team sync. If an enterprise needs group-driven Team membership, the workaround is to map IdP groups via **SAML attributes at login** instead of relying on SCIM Groups. This gap is an explicit open decision (below), not an oversight.

**`oidcProvider()` and `mcp()` are the same OAuth stack, layered.** `oidcProvider()` turns Docket _itself_ into an OAuth 2.1 / OIDC **Authorization Server**: PKCE, the `oauthApplication`/`oauthAccessToken`/`oauthConsent` tables, and an RFC 7591 `/register` endpoint for dynamic client registration. `mcp()` is built _on top of_ that provider — it does not stand up a second AS. It contributes the MCP-specific glue (`withMcpAuth` to gate the `/mcp` transport, `getMcpSession` to resolve the verified session inside a tool call, and the `oAuthDiscoveryMetadata` / `oAuthProtectedResourceMetadata` documents that satisfy RFC 8414 and RFC 9728). The architectural consequence is in architecture.md §"MCP" / §4 here: Better Auth is the Authorization Server and the `/mcp` endpoint is an OAuth 2.1 Resource Server, both on the same Hono deploy. Note the verification flagged in RECONCILIATION: confirm 1.6.14 actually emits `client_id_metadata_document_supported` (CIMD), honors RFC 8707 `resource→aud` stamping, and surfaces `getMcpSession().scopes`; if any is absent, a thin Resource-Server shim covers the gap.

**`stripe()`** is the eighth plugin; its full treatment (billing subject = Organization, the trial mechanism, the lifecycle pipeline Docket builds on top) is §3.

### Separate `@better-auth/*` packages vs. core plugins

These plugins do not all ship from the same import, and `@docket/auth`'s `package.json` must reflect that precisely. Three are **distinct npm packages**, each pinned at 1.6.14 alongside core: `@better-auth/passkey`, `@better-auth/sso`, and `@better-auth/scim`. Billing's `stripe()` is also a separate package, `@better-auth/stripe` (paired with `stripe@^22`, see §3). The remaining three — `genericOAuth`, `oidcProvider`, and `mcp` — come from the **core** `better-auth/plugins` export and need no extra dependency entry. The practical hazard is a version skew: if the separate `@better-auth/*` packages drift off 1.6.14 while core stays pinned, plugin internals can mismatch the core runtime in subtle ways, so every one of them is pinned to the identical 1.6.14 and listed explicitly rather than relying on a transitive resolution.

### Schema generated into `@docket/db`

Better Auth would normally own its own schema island; here it does not. We run its CLI — `npx @better-auth/cli@1.6.14 generate` — to emit the Drizzle schema **into `@docket/db`**, then apply it with `drizzle-kit migrate`. `@docket/db` is the single source of truth for _all_ tables, auth tables included, which is the precondition for one repo-wide ULID `genId()` primitive and a clean `ON DELETE CASCADE` graph (architecture.md §"Schema"). Because the plugin set is wide, this generate step also lands the plugin-owned tables (the passkey credential table, the `oauthApplication`/`oauthAccessToken`/`oauthConsent` set from `oidcProvider()`, the SSO/SCIM tables, and the Stripe `subscription` table keyed by `reference_id = organization.id`). Sequencing follows the build spine `env → db → auth → api`: the auth config is authored, the schema is generated into `@docket/db`, and migrations run _before_ `apps/api` mounts the handler.

### One `/api/auth` owner: Hono

There is exactly one mount point for `/api/auth/*`, and it is the **Hono service `apps/api`**, not any Next app. This is the resolved decision in RECONCILIATION (the recommendation in the original plan, now confirmed): the Next apps are pure clients that consume Better Auth via `createAuthClient` (with `passkeyClient()` for the WebAuthn flows). Inside Hono the handler is mounted as `app.on(["POST","GET"], "/api/auth/*", (c) => auth.handler(c.req.raw))`, and **CORS middleware must be registered first**, exposing the `Authorization` and `WWW-Authenticate` headers (the latter is required for the MCP Resource Server's 401 challenge). The same-origin story that keeps the session cookie first-party — `apps/web` rewriting `/api/*` to `apps/api` at the Vercel edge — is described in architecture.md §"System" / §"API"; CORS-with-credentials is the fallback only if the deployables must be split. Concretely this means `BETTER_AUTH_URL == API_URL` (env spec §0/§1.2): the auth base URL, the OIDC issuer, and the MCP resource base are all the same origin, which is what prevents `redirect_uri_mismatch` across the OAuth callbacks and lets passkey RP-ID = the shared apex (`docket.app`) work across subdomains.

### Open decisions

Four auth questions remain open and are tracked in §7. Most are now resolved in RECONCILIATION but listed here for traceability: the **canonical `/api/auth` owner** (resolved → Hono/api); **`oidcProvider()` + `mcp()` together vs. `mcp()` alone** (resolved → both, since `mcp()` builds on the provider); the **SCIM-Groups gap** (no plugin support → SAML-attribute group mapping if needed); and the **`trustedProviders` account-linking trust policy** (which providers may auto-link by email — a live security-boundary decision to keep minimal).

## 3. Billing & Data Lifecycle (Stripe)

Docket bills through Stripe Billing subscriptions, mediated by the Better Auth **`@better-auth/stripe`** plugin (SDK pinned `stripe@^22`, API version `2026-03-25.dahlia`). The single most consequential decision here is _who_ gets billed: the billing subject is the **Organization**, not the User. The plugin is configured with `referenceId = Organization.id` and the `authorizeReference` hook gating every billing mutation, so each org owns exactly one Stripe customer and one subscription, and a member can only act on billing for an org they are actually authorized to administer. This falls directly out of the identity model in §2 — a User is a global account that can belong to many orgs or none, so the User is the wrong unit to charge; the Organization is the tenant boundary, the thing that has a subscription, a trial clock, and ultimately a deletion fate. Per RECONCILIATION the Better Auth _organization plugin_ is **off** (Docket owns its custom `organization`/`actor` shape), but the Stripe plugin's `subscription` table is still keyed by `reference_id = organization.id`, and `authorizeReference` is what enforces that a caller may only touch the subscription for an org where their human `Actor` holds the right capability. This is also why every server-only Stripe secret lives in `apps/api` alone (see architecture.md §"Secrets ownership") — there is one place that holds `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET`, and one surface to audit for cross-tenant billing bugs.

### The plugin boundary: what Stripe owns vs. what we build

The clean division of labor is the heart of this section. **The `@better-auth/stripe` plugin owns the Stripe-facing machinery**: it creates the customer on signup, maintains the `subscription` table and its status mirror, processes the four core webhooks (`checkout.session.completed`, `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`), exposes upgrade/cancel/restore/`billingPortal` operations, and enforces one-trial-per-account. We do not re-implement any of that.

**What the plugin has no concept of — and what we therefore build ourselves — is the data lifecycle:** what happens to a tenant's _data_ after billing ends. Stripe will tell us a subscription was canceled; it will never grant a grace period, generate an export, or delete a row. So Docket layers a state machine directly on the `organization` row via three columns: `lifecycle_state` (the enum `{trialing | active | past_due | export_window | pending_deletion | deleted}`), `export_ready_at`, and `delete_after_at`. (RECONCILIATION fixes these as the canonical snake_case columns on `organization`, alongside the rule that the Stripe `subscription` table is keyed by `reference_id`; the engineering surface refers to them as `lifecycleState`/`exportReadyAt`/`deleteAfterAt`.) The architecture doc carries the full state diagram and the narrative of the transitions in architecture.md §3 ("Organization Data-Lifecycle State Machine") and §"Runtime & Lifecycle Flows" — this section is the implementation contract, not a re-drawing of it.

The transitions are driven by Stripe events but never _trusted_ from them. When a trial ends without payment, or an `active` org's `invoice.payment_failed` drives it through `past_due` to a terminal failure, the org moves to `export_window`: we set `export_ready_at`, stamp `delete_after_at = now + 14d`, generate a downloadable export of the org's entire work layer, and email the operator the link. The 14-day window is a deliberate grace period, not a courtesy — it is the difference between "billing lapsed" and "data destroyed," and it is the one place reactivation must be able to intervene.

### The idempotent cron sweep, and why it reconciles rather than trusts webhooks

Stripe webhooks are **neither guaranteed to arrive nor guaranteed to arrive in order**. A naive design that flips `lifecycle_state` purely in webhook handlers would strand orgs in wrong states on a dropped event and risk double-deletion on a replayed one. So the actual deletion is performed by an **idempotent cron sweep** — a Vercel Cron call to `/lifecycle/sweep`, guarded by `CRON_SECRET` sent as `Authorization: Bearer` (the secret bootstrap generates with `openssl rand -hex 32` and writes to Vercel; see env-and-bootstrap.md §1.7). The sweep's two non-negotiable properties are:

- **Idempotency by reconciliation.** For each org it loads the _live_ Stripe subscription state and reconciles `lifecycle_state` against that truth, rather than believing whatever webhook last fired. Running it twice produces the same result; a lost or out-of-order `customer.subscription.updated` cannot corrupt the state machine because the sweep recomputes from Stripe each pass.
- **Holds are respected.** The sweep **skips any org under an active `LifecycleHold`** (the operator-plane escape hatch — e.g. a legal hold — that pauses the trial→export→delete pipeline without touching billing state, per §5 service-admin layer).

When the sweep does delete, it advances `pending_deletion → deleted` and relies on `ON DELETE CASCADE` from `organization` through every `organization_id` FK (the cascade-as-lifecycle-policy described in architecture.md §"containment vs. association"), plus an application-level purge job for artifacts that do not live in our Postgres — the Stripe customer and any external connector state. **Reactivation cancels a pending deletion:** paying or restoring at any pre-deletion stage returns the org to `active` and clears `delete_after_at`, which is precisely why the transitions can't be one-directional. Critically, this lifecycle gate runs _upstream_ of the permission engine — a frozen org is a 402/403 concern resolved before `canActor` is ever consulted, so authorization stays purely about capability while lifecycle stays purely about whether the tenant may transact at all.

Webhooks beyond the plugin's four are handled through the plugin's `onEvent` hook: `customer.subscription.trial_will_end`, `invoice.payment_failed`, `invoice.paid`, and `invoice.payment_action_required`. These feed the state machine (e.g. `trial_will_end` warns ahead of the trial cliff; `invoice.payment_failed` drives `active → past_due`), but the sweep's reconciliation remains the source of truth even for them.

### Trials and the embedded Checkout

The trial uses `plan.freeTrial.days: 14`. As the Hard Constraints (§0) call out explicitly, this is **named** "legacy trial" but is the **supported, non-deprecated** path — and the only one compatible with both Stripe Checkout and the Better Auth plugin. The newer Trial Offer API (preview) is Checkout-incompatible, so adopting it would mean abandoning Checkout entirely; we keep the supported trial path on purpose. The default flow is **embedded Checkout** (rendered in the product app via `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`), not a redirect to a hosted page.

### The tier-dependent card gate

Whether a card is required is a **product decision keyed to tier**, not a global setting. A Personal/solo org requires **no card** — a single user gets a frictionless start with no payment method at all. A card is required only when the user crosses into shared territory: **creating a shared/team org, or inviting members into one.** This is why bootstrap creates no price for the personal tier (env-and-bootstrap.md §1.4 note) — the Team price set is what gates shared orgs and invites, and the personal path never touches Stripe Checkout. The no-card-vs-card-required _default_ at the trial boundary remains an open decision (below).

### Test/live parity, per-mode webhook secrets, and `lookup_key`

Billing obeys the same dev-mirrors-prod contract as everything else (§0, env-and-bootstrap.md §4): **test mode mirrors live mode** with identical variable names and only the values differing. `STRIPE_SECRET_KEY` is `sk_test_…` in dev and `sk_live_…` in prod; `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` is `pk_test_…` vs `pk_live_…`. There is no `DEV_`/`PROD_` branching and no `if (NODE_ENV === 'development')` switch of which webhook or callback is wired — only the value behind each name changes.

Webhook signing secrets are **per-mode and per-endpoint**. The endpoint path is fixed at `${API_URL}/api/auth/stripe/webhook`. In **dev**, `STRIPE_WEBHOOK_SECRET` is the `whsec_…` printed by `stripe listen --forward-to localhost:8787/api/auth/stripe/webhook` (captured via `stripe listen --print-secret`); in **prod** it is the signing secret of the real endpoint that bootstrap registers with `stripe webhook_endpoints create`. A test secret can never verify a live signature and vice versa, so the per-mode separation is a correctness requirement, not a convenience.

Plan→price resolution **never hardcodes price IDs**. The primary mechanism is a stable, mode-agnostic **`lookup_key`**: `DOCKET_PRICE_LOOKUP_TEAM` (set to `team_monthly` when bootstrap creates the price with `--lookup-key team_monthly`) and the optional `DOCKET_PRICE_LOOKUP_TEAM_ANNUAL` (`team_annual`). Because the lookup key is the same string in both modes, the same code resolves it to whichever active price exists in the current mode. The explicit per-env price ID `STRIPE_PRICE_TEAM` (`price_…`) exists only as a fallback/override for when lookup-based resolution is disabled — and even it is sourced from env, never inlined. An optional `STRIPE_BILLING_PORTAL_CONFIG_ID` selects a non-default Customer Portal configuration when one is used.

### Open decisions

Several billing questions remain open and are deferred to the billing phase (consolidated in §7 and RECONCILIATION's "Billing phase"):

- **No-card vs. card-required** as the trial _default_.
- **`missing_payment_method`** behavior on trial end: cancel the subscription vs. pause it.
- **`past_due` terminal state:** treat as `unpaid` (recommended) vs. `canceled`.
- **Cron infrastructure** for the sweep (Vercel Cron is the working assumption).
- **Export artifact format and scope** — what the downloadable work-layer export contains and how it's stored (`EXPORT_BUCKET_URL`/`EXPORT_BUCKET_TOKEN`, e.g. Vercel Blob).

## 4. MCP Remote Server (spec 2025‑11‑25)

Docket exposes its entire work layer to agents not as a bespoke API‑key surface but as a standards‑compliant OAuth 2.1 **Resource Server (RS)**, with Better Auth — mounted at `/api/auth/*` in the same `apps/api` deploy — acting as the **Authorization Server (AS)** through its `oidcProvider()` + `mcp()` plugins. There is a single `/mcp` endpoint in the Hono app, speaking the **Streamable HTTP** transport (`POST` for JSON‑RPC, which may upgrade to SSE; `GET` for the server→client SSE stream). The strong recommendation, which the `mcp-surface.md` spec elevates to a decision, is to mount the official `@modelcontextprotocol/sdk` `StreamableHTTPServerTransport` under a Hono route rather than hand‑rolling SSE framing, session handling, and resumability — the SDK also enforces the `MCP-Protocol-Version: 2025-11-25` header and rejects unknown versions with HTTP 400 for us, and the deprecated two‑endpoint HTTP+SSE transport is forbidden outright. The structural picture — the transport → RS → capabilities layering, the well‑known routes, and the data‑layer fan‑out — lives in **architecture.md §"MCP Server Architecture"** (including the full block diagram); this section is about _why the splits are drawn where they are_, the exact configuration, the sequencing, and the failure modes that have to be closed before this surface can ship.

### The MUSTs, and why each one exists

Five protocol‑level MUSTs are non‑negotiable, and each closes a specific attack or confused‑deputy class. **Origin validation** runs _before any auth work_: requests whose `Origin` is not in the allowlist (`https://app.docket.*`, `https://*.docket.*`, plus configured client origins) are rejected outright, defeating DNS‑rebinding against a listener bound to the platform host. A **Bearer token is required on every request**, in the `Authorization` header and never the query string; a missing or invalid token is a 401. Tokens are **audience‑bound per RFC 8707**: the AS is asked to mint against `resource=<canonical /mcp URL>` (`https://api.docket.app/mcp`, a per‑env value), and the RS rejects any token whose `aud` does not equal that canonical RS URI — this is what stops a token a client obtained for some _other_ resource from being replayed against Docket. The RS **serves its own discovery**: Protected Resource Metadata per **RFC 9728** at both `/.well-known/oauth-protected-resource` and the `/mcp` sub‑path form (`/.well-known/oauth-protected-resource/mcp`), plus AS metadata per **RFC 8414** — and the latter MUST advertise `code_challenge_methods_supported: ["S256"]`, because conformant clients refuse to proceed without it, so that field is load‑bearing rather than decorative. Finally, **no token passthrough**: the RS MUST NOT accept tokens minted for GitHub/Drive/Linear, and MUST NOT forward the client's inbound token to any downstream system. When `link_external` or `start_connector_link` needs to reach a provider, it resolves the org's own Integration credentials (`Integration.connection.credentials_ref`, data‑model §5), never the caller's token. Together, audience binding and no‑passthrough are the two properties that keep a multi‑tenant RS from becoming a confused deputy.

The validation chain itself is concrete and ordered. `withMcpAuth(auth, handler)` wraps `/mcp`; inside it, `auth.api.getMcpSession({ headers })` yields `{ accessToken, userId, scopes, clientId }`, and the RS then enforces, in sequence: Bearer present → `aud` == canonical RS URI → `iss` == Docket AS issuer → scope sufficient for the requested operation → principal resolution (`sub`/`userId` → `User`, then the human `Actor` row for `(user_id, organization_id)`) → granular `Grant` cascade. Failures surface as the two prescribed `WWW-Authenticate` challenges: a 401 carries `Bearer resource_metadata="…/oauth-protected-resource/mcp"` pointing the client at PRM, while a runtime 403 carries `error="insufficient_scope"` naming both the scope this operation needs and the relevant already‑granted scopes — the mechanism by which a read‑only agent escalates (a write tool returns 403 → client runs step‑up authorization → re‑calls with `work:write`).

### Capabilities: support as much as the spec offers

Docket advertises a deliberately wide capability set on `initialize` (`protocolVersion: "2025-11-25"`): **tools**, **resources + resource templates**, **prompts**, **completion**, and **logging**, surfacing `listChanged` for tools and both `subscribe` and `listChanged` for resources, and honoring the lifecycle utilities — pagination (`cursor`/`nextCursor`), progress (`notifications/progress`), cancellation, and `ping`. The `tools.listChanged: true` advertisement is meaningful because the visible tool set is principal‑ and org‑aware: a token lacking `agents:run` does not see the agent tools, and unlinked connectors hide the relevant `link_external` subjects, so the set genuinely changes mid‑session when grants or connectors change. Two capabilities are explicitly _optional_ and adopted behind flags: the experimental **Tasks** utility (`tasks.requests.tools.call`), a good fit for long agent runs and large views via `trigger_agent_session`/`run_view`, gated behind `MCP_TASKS_ENABLED` because the spec area is still churning; and **URL‑mode elicitation**, used to hand off connector OAuth to a browser via `start_connector_link`'s `authorize_url` without ever passing a Docket token to the provider. (Note that the spec defers `prompts` from v1 — no server‑defined prompt templates yet — which is recorded so implementers do not add it speculatively.)

### Entity → tool/resource mapping

The tool/resource split maps onto the read/write distinction. **Reads are resources/templates** addressed under `docket://{org}/{type}/{id}` — Project, Program, Initiative, Task, Cycle, Team, Update, Agent, Session, plus saved Views — where `{org}` is the Organization **slug** (Personal space = `personal`; Hub‑scoped reads use the literal `hub`, e.g. `docket://hub/today`). The scheme is a custom, RFC 3986‑compliant, **non‑fetchable** URI: clients MUST read through `resources/read` (which returns a Zod‑validated JSON read DTO), and the mutation tools additionally hand back an `https://app.docket.app/…` deep link for human navigation. **Mutations are tools** — `create_task`, `update_task`, `move_task`, `set_task_assignee`/`set_task_delegate`, `add_task_dependency`/`remove_task_dependency`, `post_update`, `link_external`, `trigger_agent_session`, `approve_action`/`reject_action`, `run_view`, and the rest — each authored with Zod input _and_ output in `@docket/types`, converted to JSON Schema 2020‑12, populating both `structuredContent` and a serialized‑JSON `TextContent` block for backwards compat. Every tool sets all four `ToolAnnotations` (`readOnlyHint`/`destructiveHint`/`idempotentHint`/`openWorldHint`) **explicitly** rather than leaning on the SDK defaults — notably `destructiveHint` defaults to `true`, and only `link_external` and `trigger_agent_session` (which touch external systems) set `openWorldHint: true`; everything else is closed‑world. The one nuance: `run_view` and `search` are _reads_ exposed as **tools**, not resources, because they take rich query arguments that don't fit a URI — they keep `readOnlyHint: true` and need only `work:read`. (Per RECONCILIATION, the shared `Id` primitive is repo‑wide **ULID `text`**; although the draft tool schemas write `z.string().uuid()`, MCP must consume the branded `Id` from `@docket/types` and not assert UUID shape — the addressing differs from REST but the id primitive is identical.)

Authorization is **two‑layered**, and conflating the layers is the most likely implementation mistake. The token carries one of four flat, _global_ scopes that gate only the capability class:

| Scope             | Capability class                                            | RS‑enforced per‑org grant                                   |
| ----------------- | ----------------------------------------------------------- | ----------------------------------------------------------- |
| `work:read`       | Read any visible work‑layer entity                          | `view` on the resolved resource                             |
| `work:write`      | Create/update/move/assign/comment/post‑update/run‑view/link | `contribute`/`assign`/`comment`/`manage` per tool           |
| `agents:run`      | Trigger sessions; approve/reject agent actions              | `manage` on target + approver‑eligibility                   |
| `connectors:link` | Link external resources / initiate connector OAuth          | `manage` on Integration + `contribute` on the linked entity |

A scope is **necessary but never sufficient**: every call _also_ resolves the caller's `Actor` in the target org and evaluates the granular `Grant` cascade (`view`/`comment`/`contribute`/`assign`/`manage`), so a `work:write` token still earns a 403 if the principal lacks `contribute` on that specific task. Scopes are global rather than org‑qualified (`work:write:org_<id>`) on purpose — the product model is one global `User` with org access via `Actor` membership, so per‑org scopes would bloat the consent screen and force re‑consent on every new membership; instead the **org is a call argument** (the `org` slug), authorized at execution time.

### Multi‑tenant safety: trust only the verified `sub`

The single most important RS invariant is that org and user context derive **only** from the verified token's `sub` (→ `User`, then the human `Actor` for `(user_id, organization_id)`) and the granular permissions on those rows — _never_ from anything the client asserts. The `docket://` URI's embedded slug is used for _addressing_ but is never trusted for _access_; if no `Actor` membership row exists for the requested org, the call is a 403. To avoid leaking entity existence to unauthorized callers, a missing grant on a `resources/read` returns not‑found (`-32002`), not forbidden.

### Open decisions and the Better‑Auth‑1.6.14 verification risk

A cluster of decisions is recorded as open: single AS vs per‑org external AS (the spec resolves to a single AS, multi‑tenant RS); **CIMD vs DCR vs both** (resolved: CIMD primary via `client_id_metadata_document_supported: true` with an SSRF guard + domain‑trust allowlist, DCR at `/register` per RFC 7591 kept as a MAY‑level fallback since 2025‑11‑25 downgraded it, first‑party clients pre‑registered); whether to adopt **Tasks** now vs defer (deferred behind `MCP_TASKS_ENABLED`); stateful + resumable session vs stateless; `docket://` vs fetchable `https://` resource URIs (resolved to the non‑fetchable custom scheme); and the exact MCP scope set (`work:read`, `work:write`, `agents:run`, `connectors:link`; resolved to per‑user‑global rather than per‑org). On the session‑store question the engineering tie‑breaker overrides the spec: `mcp-surface.md` proposes a Redis‑backed event store with `Last-Event-ID` resumability, but **RECONCILIATION pins v1 to Postgres** (the `idempotency_key` table, 24h TTL) plus plain SSE for live session streaming — one mechanism, no Redis — deferring the Redis resumable store and a long‑lived Vercel Fluid Compute host to the MCP phase (see architecture.md §"MCP Server Architecture").

The hard contingency that makes this whole surface _provisional_ is a Better Auth **1.6.14** verification gate, and it must be settled before build, not assumed. Three behaviors are unconfirmed: (1) **CIMD** — that the AS natively emits `client_id_metadata_document_supported` and validates URL‑form `client_id`s (matching `client_id` === URL exactly and checking `redirect_uris`); (2) **`aud` stamping** — that the AS actually binds the token `aud` from the RFC 8707 `resource` parameter; and (3) **`getMcpSession().scopes`** — that the helper surfaces the granted scopes the RS needs for step‑up enforcement. If any of these is absent, the fallback is well‑defined: a thin CIMD shim in the RS, and for audience binding a fall back to issuer‑plus‑client binding with an explicit resource allowlist. The verification flow itself is a declared e2e flow (engineering plan §6): connect MCP client → discover PRM/AS → CIMD register → consent → call a read tool → step‑up to write, plus a `trigger_agent_session` → subscribe to the session resource → `approve_action` flow. The env contract that backs all of this is validated in `@docket/env` with dev mirroring prod: `MCP_CANONICAL_URL`, `MCP_ALLOWED_ORIGINS`, `MCP_SESSION_STORE_URL`, `MCP_TASKS_ENABLED`, plus the shared `BETTER_AUTH_URL`/secret/DB vars.

## 5. Data Model

The data layer is Postgres accessed through Drizzle, with `@docket/db` as the single SQL owner for the whole monorepo — app tables and Better Auth's CLI-generated tables live and migrate together from one package (see architecture.md §Monorepo Structure). The authoritative, implementation-grade schema — every `pgTable`, every index, every FK and `$type<>` — is `specs/data-model.md`; architecture.md §Domain Model carries the conceptual _why_ (the containment-vs-association narrative, why `Program` is its own entity, identity≠membership, the DAG integrity problem). This section is the engineering-plan view: the modeling decisions that constrain the build, followed by the entity field reference those decisions produce. Read it alongside both.

Five decisions shape everything below, and they are not independent — each one falls out of the product model and forces the next.

**Tenant key = `organization_id`, in the row and in the index.** Every work-layer and org-scoped table carries `organization_id text NOT NULL REFERENCES organization.id ON DELETE CASCADE`, and that column is the _leading_ column of the table's primary index (composite `(organization_id, …)` indexes throughout), so every tenant-scoped query is index-prefixed by tenant. The hard rule that the data-access layer — Hono RPC handlers and MCP tools alike — must inject `organization_id` from the verified token/session context and **never** from a client-asserted field is the central safety invariant (architecture.md §High-Level System Architecture, §4 here). The few genuinely cross-org tables — `hub`, `user`, `daily_plan_item`, `notification` — deliberately omit `organization_id` from the row and instead reference an org through a nested column (e.g. `daily_plan_item.ref_organization_id`, `notification.organization_id` as "which org it's from"); they are the Hub cockpit's union-across-tenants surface, never a cross-tenant join.

**IDs are text ULIDs, repo-wide, one generator.** Every primary key is `text("id").primaryKey().$defaultFn(genId)` where `genId()` lives once in `@docket/db/src/id.ts` and returns a 26-char sortable `ulid()`. This is a RECONCILIATION tie-breaker, not a preference: Better Auth's CLI emits `text("id").primaryKey()` for `user`, so any FK pointing at it (notably `actor.user_id`) must be `text` too — **Postgres-native `uuid()` is forbidden for app tables** because mixing `uuid` and `text` fractures the FK graph. Better Auth's `advanced.database.generateId` is wired to the same `genId` so even auth-minted rows obey it, and `@docket/types` exports one branded `Id` (ULID-shaped) consumed by both the REST and MCP schemas — addressing differs (`/v1/orgs/:orgId` vs `docket://{slug}/…`) but the id primitive is identical, so MCP tool schemas must **not** assert `z.string().uuid()`. (ULID vs cuid2 is settled as ULID; never mix the two generators.) All timestamps are `timestamptz` (`{ withTimezone: true }`); `created_at`/`updated_at` default `now()` with `updated_at` driven by `$onUpdate`; `archived_at` nullable marks soft-delete (active = `archived_at IS NULL`), with `completed_at`/`canceled_at` on lifecycle entities. Stable cross-table value sets (status/health/priority/kind) are declared once as `pgEnum`s in `@docket/db/src/enums.ts`; single-use sets use inline `text(col, { enum: [...] })`. Structured blobs (`preferences`, `vocabulary`, `agent_guidance`, `connection`, `provenance`, `workflow_states`, `metadata`) are `jsonb().$type<T>()` with a Zod-derived `$type`.

**Containment cascades; association sets-null or goes m2m.** This is the load-bearing distinction, encoded directly in the FK graph: a hard parent the child cannot outlive is a `NOT NULL` FK with `ON DELETE CASCADE` (org→everything, program→project, project→milestone, team→cycle, agent→agent*session, agent_session→session_activity, task→subtask via `parent_task_id`), while a soft, optional link is a nullable FK with `ON DELETE SET NULL` (`project_id`, `program_id`, `milestone_id`, `cycle_id`, `assignee_id`, `delegate_id` on a task; `created_by`/`author_id`/`owner_id`/`lead_id` everywhere, so history survives an actor's departure) or a dedicated m2m join table. The cascade choice \_is* the lifecycle policy: deleting an organization obliterates its entire work layer in one statement (the lifecycle purge), whereas detaching a task from a project demotes it to Triage intact. One deliberate exception: `task.team_id` is `NOT NULL ON DELETE RESTRICT` — every task always belongs to a team, and a team can't be deleted out from under live tasks.

**Subjects that point at many types are polymorphic columns, not FKs.** `Update`, `Comment`, `Notification`, `audit_event`, and `ImpersonationSession` use the `(subject_type <pgEnum>, subject_id text)` pattern with **no DB FK** (a polymorphic column cannot reference one table); integrity is app-enforced and the pair is indexed `(organization_id, subject_type, subject_id)` for fan-in lookups.

**`task_dependency` is a directed acyclic, org-wide, cross-project graph — and no single Postgres constraint can enforce that.** A task may be blocked by any other task in the same organization (across projects/programs), surfaced both as "blocking" and "blocked by," and the graph must stay acyclic. Three layers defend it: `CHECK(blocking_task_id <> blocked_task_id)` kills the trivial self-loop, the composite `PRIMARY KEY (blocking_task_id, blocked_task_id)` forbids duplicate edges, and the real guard is application-level — before INSERT, a recursive-CTE reachability query runs inside a `SERIALIZABLE` (minimum `REPEATABLE READ`) transaction and rejects the edge if `blocked` can already reach `blocking`. The query is `WHERE organization_id = $org`, so dependencies are index-prefixed and structurally can never cross a tenant boundary even though they freely cross project boundaries; serializable isolation is what closes the race where two concurrent edges each look acyclic alone but together form a cycle (a deferred constraint trigger is the optional defense-in-depth).

With those decisions fixed, the entity reference follows. The common "auditable entity" mixin — `auditColumns()` — spreads `id`, `organization_id`, `created_by` (→ `actor`, `set null`), `created_at`, `updated_at`, `archived_at?` into work entities.

**Identity / container layer.**

- **`user`** (Better Auth core, **global, not org-scoped**): `id, name, email, avatar?, auth`; the account that persists at zero memberships (architecture.md §Auth & Identity Topology); 1:1 with a `hub`. Docket's `actor.user_id` FKs here.
- **`hub`** (personal command center, 1:1 user): `id, user_id (unique), name?, preferences`. No `organization_id` — it gathers orgs via the user's `actor` rows and owns the Personal space, `daily_plan_item`s, and `notification`s.
- **`organization`** (shared tenant / context boundary): `id, name, slug (unique), avatar?, is_personal, vocabulary` (the skin), `agent_guidance`, plus the Docket-owned billing-lifecycle columns `lifecycle_state {trialing|active|past_due|export_window|pending_deletion|deleted}`, `export_ready_at`, `delete_after_at` (indexed `(lifecycle_state, delete_after_at)` for the §3 cron sweep). The Better Auth Stripe `subscription` row is keyed by `reference_id = organization.id`. Personal space = `organization{is_personal:true}` with one default team. **The Better Auth organization plugin is OFF** (RECONCILIATION): Docket owns this custom `organization`/`actor` shape, and invitations are a hand-built `invitation {id, organization_id, email, role, token, expires_at, status}` table rather than the plugin's.
- **`actor`** (org-scoped "who"): `id, organization_id, kind {human, agent, team}, display_name, avatar?, status {active, suspended}`, plus human-only nullable `user_id` (→ `user`) and `role_id` (→ `role`). A human Actor **folds in membership** — there is no separate Membership table — guaranteed one-per-person-per-org by a partial unique index `(organization_id, user_id) WHERE user_id IS NOT NULL`. Assignable iff `kind ∈ {human, agent}`.
- **`team`** (first-class within-org): `name, key, description?, workflow_states[]` (jsonb; default `{backlog, todo, in_progress, done, canceled}`), `triage_enabled, agent_guidance?`. Owns cycles and triage; explicit (not auto-created beyond the Personal-space default). Agent-guidance resolution layers org → team → per-agent (team overrides `organization.agent_guidance`, agent overrides team). A normalized `team_member {team_id, actor_id, organization_id}` join is the "people in a team" set.

**Work layer.**

- **`initiative`** (pure theme, no work inside): `name, description, owner_id, status {active, completed}, target_date?, health?`; m2m to projects and programs; has updates.
- **`program`** (ongoing ops): `name, description, owner_id, status {active, paused, archived}` — **deliberately no `completed`**, because a Program cannot be marked done — `health?`; contains projects + tasks; m2m to initiatives; has updates.
- **`project`** (bounded): `name, description, lead_id, program_id? , status {planned, active, completed, canceled}, health?, start_date?, target_date?, team_id?`; contains tasks; has milestones (own table); m2m to initiatives; has updates.
- **`cycle`** (team-scoped): `team_id, number, name?, starts_at, ends_at, status {upcoming, active, completed}`, unique `(team_id, number)`.
- **`task`** (atomic unit): `title, description, team_id` (NOT NULL, RESTRICT), `state` (free-text key into the team's `workflow_states`, no global FK), `priority {none,urgent,high,medium,low}, assignee_id?`→Actor(human|agent), `delegate_id?`→agent, `project_id?, program_id?, milestone_id?, cycle_id?, parent_task_id?` (subtask tree, cascade), `estimate?` (optional unit, off by default), `due_date?, labels[]` (via `task_label`), `completed_at?, canceled_at?`. Triage = `project_id IS NULL AND program_id IS NULL` on a triage-enabled team (derived, not a column). **Provenance** inline: `provenance_source {native|linked}, source_integration_id?, external_id?, external_url?, provenance_sync_mode {import|mirror}`, with a partial unique `(source_integration_id, external_id) WHERE external_id IS NOT NULL` for idempotent import/mirror.
- **`milestone`** (dated checkpoint, conceptually an attribute of a Project but a **real row** so `task.milestone_id` has a valid FK target): `id, organization_id, project_id (cascade), name, target_date?, sort`.

**Agents & sessions** (thin agent identity; the Session is the real entity).

- **`agent`** (1:1 with an `actor{kind:'agent'}`): `connection {endpoint, protocol, credentials_ref}` (provider — Athena/Claude/Codex), `approval_policy {suggest | act_with_approval | autonomous}`, `accountable_owner_id`→Actor, `guidance?`, `approval_routing?`. Two orthogonal dials: _what it may touch_ = `grant` rows to the agent's actor (the same permission system as humans), _whether writes need sign-off_ = `approval_policy`. The approver for `act_with_approval` defaults to the task's assigner/delegator, overridable per Org/Team via `approval_routing`. **Compute/cost/telemetry are NOT stored — the provider owns them.**
- **`agent_session`** (Docket-hosted episode; TS export `agentSession`, table name **`agent_session`** to avoid colliding with Better Auth's login `session`): `id, organization_id, agent_id, task_id?, trigger {assignment, delegation, mention}, status {pending, running, awaiting_input, awaiting_approval, completed, failed, canceled}, external_run_ref?, started_at, ended_at?`.
- **`session_activity`** (per-session stream, distinct from the org-wide audit feed): `id, session_id, type {thought, action, response, elicitation, error}, body`, plus `approval_status {proposed → approved/rejected → applied}` and `applied_audit_event_id?` (only meaningful when `type='action'` under `act_with_approval`). `response`/`elicitation` activities are mirrored into the task `comment` stream by posting a comment authored by the agent's actor.

**Cross-cutting.**

- **`role` + `grant`** (granular permissions; agent scopes use the same system): a `role` is an org-scoped named capability bundle (`Owner/Admin/Member/Guest` seeded `is_system`, plus custom), referenced by `actor.role_id`. A `grant {subject_kind: actor|role, subject_id, resource_kind: organization|team|program|project|task, resource_id, capabilities[] {view, comment, contribute, assign, manage}, visibility {public, private}}` **cascades down containment**, overridable lower (resolution is app logic). Agent scopes are simply grants whose subject is an agent actor. Default visibility is role-dependent: public to org members, but **guests are grant-only**.
- **`update`** (status post): `id, organization_id, author_id, subject {project|program|initiative}` (polymorphic), `health {on_track,at_risk,off_track}, body, created_at`. The latest update sets its subject's current health.
- **`daily_plan_item`** (Hub-scoped, personal, cross-org): `id, hub_id, date, ref_organization_id + ref_task_id` (the cross-org `task_ref` pair), `sort, status {planned, done}, timebox?`. No `organization_id` on the row.
- **`notification`** (cross-org; backs the Hub Inbox): `id, user_id, organization_id` (which org it's from), `type {approval, mention, assignment, status, comment, blocker, due}, subject` (polymorphic), `read_at?, acted_at?, created_at`. The Inbox is the unread/unacted set across a user's orgs — distinct from the Activity feed (action vs. awareness).
- **`integration`** (org-scoped): `id, organization_id, provider, pattern {migration, connector}, roles[] {work, context, signal, time, code}` (pg enum array), `connection {credentials_ref, account}, status {connected, error, disconnected}, config, sync_mode {import, mirror}`.
- **`label`** (org-scoped, optional `team_id`): `id, organization_id, name, color, group?`; m2m with tasks via `task_label`.
- **`comment`** (polymorphic subject): `id, organization_id, author_id` (Actor), `subject {task|project|program|initiative|cycle|update}, body, parent_comment_id?, created_at, edited_at?`. Agents post comments **as their actor** — this is how a session's `response`/`elicitation` reaches the task comment stream.
- **`audit_event`** (org-wide Activity feed / audit log; a separate table from `session_activity` despite the shared product word "activity"): `id, organization_id, actor_id` (principal), `initiator_id?` (the agent set-in-motion-by), `subject` (polymorphic), `type {created, state_changed, assigned, commented, …}, metadata {before/after}, created_at`.

**Join / edge tables** (implied by the model — they are real tables): `milestone` (above); `initiative_project` and `initiative_program` (the m2m theme links); `task_label` (Task↔Label m2m); and `task_dependency {blocking_task_id, blocked_task_id, organization_id, created_at}` — the directed, **acyclic**, org-wide, cross-project blocks edges enforced as described above.

**Service-admin layer** (service-level, NOT per-org tenant data; these reference orgs/users but belong to Docket-the-business and are excluded from the org-lifecycle CASCADE purge — backs §8.9 of the product plan):

- **`staff_user`** `{id, user_id (unique), role {support, finance, superadmin}}` — Docket operators.
- **`impersonation_session`** `{id, staff_user_id, target_user_id|target_org_id, reason, started_at, expires_at}` — time-boxed, reason-logged "View as," with a `CHECK` that at least one target is present.
- **`lifecycle_hold`** `{id, organization_id, reason, placed_by (→ staff_user, RESTRICT), placed_at, released_at?}` — pauses the trial→export→delete pipeline (e.g. a legal hold); the lifecycle cron sweep MUST skip orgs with an active (un-released) hold.
- **`operator_audit_event`** `{id, staff_user_id, action, target_user_id?, target_org_id?, metadata, created_at}` — operator actions, distinct from the per-org `audit_event` feed.

The `@docket/db` package layout, the Better Auth CLI generation/migration process, the Neon pooled-vs-unpooled connection split, and the full entity-relationship summary are detailed in `specs/data-model.md` §2, §9, §10 and architecture.md §Domain Model.

## 6. Testing & Infrastructure (Playwright)

End-to-end testing lives in a dedicated **`apps/web-e2e`** workspace member with its own **`playwright.config.ts`**, pinned to **Playwright ≥ 1.60**. Keeping the e2e suite in its own package (rather than co-locating specs inside `apps/web`) is deliberate: the harness has its own dependency graph, its own fixtures, and its own CI lifecycle, and it must be buildable alongside the foundation phases without dragging the app's build into the test toolchain (see architecture.md "Monorepo Structure"; the e2e-package location is itself flagged as an open decision in §7). The browser matrix is **Chromium-only** by default — the WebAuthn virtual-authenticator path we depend on for login is most reliable there, and a single browser keeps shard counts and film-render time tractable; widening the matrix is the second half of that §7 open decision.

### Two capture modes — lean PRs vs. the film run

Every spec runs in one of two modes from the _same_ source, and the split exists because the suite serves two masters that want opposite things from artifacts. On a pull request we want speed and signal: failures must be debuggable, but a green run should leave almost nothing behind. So the **lean PR mode** configures `trace: on-first-retry`, `screenshot: only-on-failure`, and `video: retain-on-failure` — artifacts materialize only when something breaks, so the common (passing) path pays no capture cost. The opposite need is _demonstration_: we want canonical, marketing-grade recordings of every key flow. That is the **`CAPTURE_ALL=1` film run**, where video, screenshot, and trace are all forced ON. The same spec files re-run under this env flag, driven by a scheduled **`e2e-flow-films`** GitHub workflow (not the PR pipeline), rendering at **1280×800, retina scale, reduced motion, against seeded data** so the output is reproducible and visually clean enough to drop into product/marketing surfaces. The tradeoff is intentional and one-directional: PRs never pay the film tax, and films are never gated on a PR's flakiness because they live on a schedule. Whether always-record should replace the two-mode split is an open Playwright decision (§7); the two-mode design is the current answer because it lets the same flow definitions serve both verification and demonstration without forking the specs.

### The flow-recorder fixture, the flow registry, and the coverage gate

Three pieces work together to make flows self-documenting _and_ enforced. First, a **flow-recorder fixture** wraps each logical step in `step(name, fn)`: it writes a named screenshot per step and attaches the video to the HTML report, so a flow narrates itself — a reviewer reading the report sees "sign up with passkey → create organization → create project …" as discrete, captioned frames rather than one opaque recording. This is also what makes the film run legible: each `step` boundary becomes a chapter marker.

Second, a **typed flow registry** declares the set of flows the product is _required_ to cover. The registry currently names: **signup, onboarding, create-org, connect-integration, plan-project, delegate-task-approve, cross-org-today, and invite-guest.** Because it is typed (not a doc comment), a flow id is a real symbol that specs reference.

Third, a **coverage spec** closes the loop by **failing CI when a declared flow has no corresponding spec**. This is the enforcement mechanism that keeps the registry honest: you cannot quietly ship a product surface whose required flow was never written, because the missing spec turns the coverage gate red. The first vertical slice is wired into exactly this machinery — its `flows/create-org-project-task.spec.ts` registers under flow id `create-org` (extended to cover the project- and task-creation steps), so the coverage gate enforces that the slice's verifying flow continues to exist (see build-sequence-and-slice.md §2.7). The flow-recorder `step()` boundaries in that spec are the same ones that produce the slice's screenshot-per-step and attached video.

### Auth setup: per-persona storageState and per-worker tenants

Authentication is handled by a **setup project** that logs in each persona — **owner, member, and guest** — once, and persists each authenticated browser context to a **`storageState`** file. Downstream specs then start already-authenticated as the persona they need, rather than re-running the login UI per test, which is both slow and a source of flakiness. Persona separation matters because much of Docket's behavior is permission-shaped: a guest must be provably unable to do what an owner can, and the only honest way to assert that is to run the _same_ flow under different `storageState` files.

State-mutating flows additionally get **per-worker tenants** — each Playwright worker operates against its own freshly-created Organization. This is what makes parallel execution safe: two workers both creating a Project named "Paid Launch" must not collide or read each other's rows, and per-worker tenant isolation guarantees they don't. It also doubles as a real multi-tenant assertion — a worker's session reading another tenant's data (`GET /tasks?organizationId=<other-tenant>`) must return no rows / 403, which exercises the Phase-4.5 permission boundary rather than just the UI. The per-worker tenant helper and the DB-read helper used by these assertions live in `@docket/test-utils` (see build-sequence-and-slice.md §2.7).

### The CI-safe non-passkey login problem and the CDP virtual authenticator

Docket is passkey-first, and that creates a specific testing hazard: **WebAuthn is not scriptable in a headless browser.** A real passkey ceremony depends on platform authenticator UI and user presence/verification that a headless CI runner cannot satisfy, so a naive e2e login would hang or fail. There are three conceivable escapes — inject a static test credential, expose a session-seed endpoint that mints a session bypassing WebAuthn, or drive a **CDP virtual authenticator**. The decision is settled by the tie-breaker: per **RECONCILIATION.md**, CI passkey login uses the **Playwright CDP virtual authenticator** and there is **no session-seed bypass**. The reason is that the virtual authenticator exercises the _real_ WebAuthn path end-to-end — `WebAuthn.enable` then `WebAuthn.addVirtualAuthenticator` registers a software authenticator that satisfies the ceremony deterministically — so the test proves the production sign-in path rather than a parallel shortcut around it. A session-seed endpoint would be faster but would leave the actual passkey registration/assertion path untested, which is unacceptable for an auth model whose whole premise is passkey-first (see architecture.md "Sequence — passkey-first sign-up"). The fixture configures the authenticator as:

```ts
// fixture: virtual authenticator
const client = await context.newCDPSession(page);
await client.send('WebAuthn.enable');
await client.send('WebAuthn.addVirtualAuthenticator', {
  options: {
    protocol: 'ctap2',
    transport: 'internal',
    hasResidentKey: true,
    hasUserVerification: true,
    isUserVerified: true,
    automaticPresenceSimulation: true,
  },
});
```

`isUserVerified: true` and `automaticPresenceSimulation: true` are what let the ceremony complete without human interaction; `transport: 'internal'` + `hasResidentKey: true` model a platform passkey. Note that this firmly resolves the original section's tentative "test credential / session-seed endpoint / CDP virtual authenticator" menu, and supersedes the "session-seed fallback" mention in build-sequence-and-slice.md §2.7 — RECONCILIATION is the tie-breaker and it picks the virtual authenticator outright. (The CI login mechanism nonetheless remains formally listed among the §7 open Playwright decisions; the RECONCILIATION call is the one being implemented.)

### CI: sharding, blob merge, and mocking external OAuth

In CI the suite **shards** across runners and emits the **`blob`** reporter from each shard; a merge step then combines the blobs into **one unified HTML report**, so a sharded run reads like a single coherent report rather than N fragmented ones. Videos and traces are uploaded as CI artifacts for post-mortem debugging, while **`playwright/.auth/*` is gitignored** — those files hold the persisted `storageState` sessions and must never be committed (they are per-run secrets, and stale ones would silently authenticate as the wrong persona).

The one flow that cannot use a real external dependency is **connect-integration**: it touches third-party OAuth (the `GOOGLE_*` / `GITHUB_*` / `LINEAR_*` providers). We **mock/sandbox external OAuth** for that flow rather than hitting live providers, because real OAuth consent screens are non-deterministic, rate-limited, and require human interaction — exactly the headless hazards the virtual authenticator solves for our _own_ auth. Mocking the provider lets the connect-integration flow assert Docket's side of the handshake (callback handling, account linking, token storage) deterministically, while the provider's redirect/consent is stubbed. This is distinct from the passkey case: passkey login is _our_ path and we exercise it for real via CDP; external OAuth is a _third party_ and we stand in a controlled double for it.

## 7. Open Engineering Decisions (consolidated)

- **Auth:** canonical `/api/auth` owner (Hono vs Next); `oidcProvider()`+`mcp()` vs `mcp()` alone; SCIM‑Groups gap; account‑linking trust policy.
- **MCP:** single AS vs per‑org AS; CIMD vs DCR; Tasks now vs later; stateful+resumable vs stateless; resource URI scheme; scope set & granularity.
- **Stripe:** no‑card vs card‑required trial; missing‑payment behavior; `past_due` terminal; cron infrastructure; export artifact format.
- **Turborepo:** marketing app consuming `@docket/ui` vs staying lean; Drizzle pre‑1.0 pin + v1 migration plan.
- **Playwright:** e2e package location; always‑record vs two‑mode; CI login mechanism; browser matrix (Chromium‑only recommended).
- **Product scope cut:** how much of granular permissions, multi‑org breadth, and the agent/MCP surface lands in v1.
