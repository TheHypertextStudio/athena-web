# Docket — API / Hono RPC Contract Spec

> **Area:** `api-rpc-contract` · **App:** `apps/api` (Hono 4.x) · **Companion to** the data-model spec (`docs/engineering/docket-engineering-plan.md` §5) and auth spec (§2).
>
> This spec defines the complete route inventory for `apps/api`, the Hono RPC `AppType` export contract, the validation + OpenAPI/Scalar generation strategy, and the cross-org Hub aggregation endpoints. Names are kept **identical** to the data-model entities (`Organization`, `Team`, `Initiative`, `Program`, `Project`, `Cycle`, `Task`, `Milestone`, `Update`, `Comment`, `Actor`, `Agent`, `AgentSession`, `SessionActivity`, `Integration`, `Notification`, `DailyPlanItem`, `Label`, `Grant`, `Role`).

---

## 1. Architecture & Conventions

### 1.1 App boundary

`apps/api` is a **standalone Hono 4.x service** (independent deploy; clean 12-factor env-var deploy; home of the MCP server + Better Auth mount). The Next apps (`web`, `marketing`, `admin`) consume it **type-only** via the exported Hono RPC `AppType`. No business logic lives in the Next apps; they call the API over `hc<AppType>`.

### 1.2 Compilation strategy (mandatory)

Per the eng doc §1: `@docket/db`, `@docket/auth`, and `apps/api` are **compiled** (`tsc → dist`). The Hono router is **split across many files** (one router module per resource group) so Hono RPC type inference does not cripple `tsserver`. `@docket/types`, `@docket/ui`, `@docket/env` are JIT (raw TS + `transpilePackages`). The **Hono version must be byte-identical** across `apps/api` and every consumer (pin in root catalog).

### 1.3 The RPC `AppType` export pattern

Every resource group is its own `Hono` instance built with **method chaining** (`.get().post()...`) — chaining is **required** for RPC inference; never break the chain by assigning to a `const` mid-route. Groups are mounted with chained `.route()` on the root app, and the **type of the chained result** is exported.

```ts
// apps/api/src/routes/tasks.ts
import { Hono } from "hono";
import { validator } from "hono-openapi";
import { TaskCreate, TaskUpdate, TaskList, Task as TaskOut, ListQuery } from "@docket/types";
import type { AppEnv } from "../env"; // { Variables: { session, actorCtx }, ... }

const tasks = new Hono<AppEnv>()
  .get("/", describeRoute({...}), validator("query", ListQuery), (c) => { /* ... */ })
  .post("/", describeRoute({...}), validator("json", TaskCreate), (c) => { /* ... */ })
  .get("/:taskId", describeRoute({...}), validator("param", IdParam), (c) => { /* ... */ });

export default tasks;
```

```ts
// apps/api/src/app.ts — TWO composition roots, kept apart
import { Hono } from 'hono';

// (1) The PUBLIC product API under /v1 — consumed by apps/web.
const app = new Hono<AppEnv>().basePath('/v1');
const routes = app.route('/orgs', orgs).route('/hub', hub); // cross-org Hub surfaces
export type AppType = typeof routes; // ← THE public contract

// (2) The INTERNAL staff back-office under /admin — consumed ONLY by apps/admin.
// Off /v1 so it is neither in the public AppType nor the public Scalar spec.
export const adminApp = new Hono<AppEnv>();
const adminRoutes = adminApp.route('/admin', admin); // self-gated by staffMiddleware
export type AdminAppType = typeof adminRoutes; // ← the admin contract (api.admin.*)

export { app };
```

Machine/webhook edges (Stripe webhook, provider ingest, cron, the GitHub OAuth callback)
are mounted in `server.ts` under a single non-versioned `/internal/*` umbrella — outside both
contracts and the public spec — each self-authenticated by a signature or `CRON_SECRET`. Only
the user-facing SSE stream and the binary export download stay on `/v1` outside `AppType`.

- The compiled `apps/api` package exports **`type AppType`** only (no runtime) for consumers. `@docket/types` re-exports it as the canonical contract import: `import type { AppType } from "@docket/types/api"`.
- Client: `const api = hc<AppType>(env.NEXT_PUBLIC_API_URL + "/v1", { headers, fetch })`. `strict: true` in **both** client and server `tsconfig` (RPC inference requirement).
- **Nesting convention:** org-scoped resources are mounted **under `/orgs/:orgId`** so the org tenant key is always a path param. The `orgs` router chains its children: `orgs.route("/:orgId/teams", teams).route("/:orgId/tasks", tasks)...`. This keeps `organization_id` derivation uniform and the URL self-documenting.

### 1.4 Validation (Zod, in AND out)

- Use **`hono-openapi`** (`validator`, `describeRoute`, `resolver`, `openAPIRouteHandler`) — **not** `@hono/zod-openapi`'s `OpenAPIHono`. Rationale: `hono-openapi`'s `validator("json"|"query"|"param"|"header", schema)` is a drop-in for `@hono/zod-validator` that **preserves RPC type inference on chained routers** while _also_ feeding the OpenAPI document. `OpenAPIHono` + `createRoute` breaks the `.route()` chaining inference that the AppType contract depends on.
- **Validated input** is read via `c.req.valid("json"|"query"|"param")`.
- **Output is validated too** (eng constraint "Zod in and out"): every handler returns through a shared `ok(c, schema, data)` helper that `schema.parse()`es the payload in dev/test (and on a sampled basis in prod) before `c.json(...)`. The same output schema is attached to `describeRoute` via `resolver(schema)` so the documented response and the runtime response cannot drift.
- All input schemas live in **`@docket/types`** (shared with MCP tool schemas and Next server actions). Zod v4 (`zod`), schemas authored with `.meta({ id, description })` for clean OpenAPI `$ref`s.

### 1.5 OpenAPI / Scalar generation

- Spec served at **`GET /v1/openapi.json`** via `openAPIRouteHandler(app, { documentation: { info, servers, security, tags }, ... })`. OpenAPI **3.1**. `servers` is env-driven (dev vs prod base URL) — never hardcoded.
- Interactive docs at **`GET /v1/docs`** via `@scalar/hono-api-reference`: `app.get("/docs", Scalar({ url: "/v1/openapi.json", theme: "default" }))`.
- Security schemes declared once: `bearerAuth` (Better Auth session/JWT bearer) and `mcpOAuth` (OAuth 2.1 PKCE for the MCP resource server). Each protected route declares `security: [{ bearerAuth: [] }]` plus the scope it needs via a custom `x-docket-capability` extension on the route (drives the capability table below and is asserted at runtime).
- Tags = the resource groups in §3 (one tag per resource), so Scalar renders a clean grouped sidebar.

### 1.6 Auth & tenant context middleware (applied in order)

1. **CORS** (registered first; exposes `Authorization`, `WWW-Authenticate`). Allowed origins env-driven.
2. **`sessionMiddleware`** — resolves the Better Auth session from the incoming request (`auth.api.getSession({ headers })`); on protected routes a missing/invalid session → `401`. Sets `c.var.session` (`{ user, session }`). The `/api/auth/*` mount itself is owned by this same Hono service (eng §2) — it is **out of `/v1`** and not part of the RPC AppType.
3. **`orgContextMiddleware`** (only on `/orgs/:orgId/*`) — loads the **Actor** row for `(session.user.id, orgId)` where `kind = 'human'`; 404 if no membership; sets `c.var.actorCtx = { orgId, actorId, role, capabilities }`. **Tenant key `organization_id` is ALWAYS derived from this verified context — never from the body.** This mirrors the MCP rule (eng §4: org/user from verified token `sub` only).
4. **`capabilityGuard(capability, resourceLocator)`** — per-route guard that checks the resolved `Grant`/`Role` capabilities (`view | comment | contribute | assign | manage`) against the target resource, honoring **cascade-down containment** (Org→Team/Program→Project→Task) with lower-level overrides, and `visibility {public, private}` (guests are grant-only). Returns `403` on failure.

### 1.7 Standard request/response conventions

- **IDs:** branded string types per entity (`OrganizationId`, `TaskId`, …) — opaque to clients.
- **Pagination:** cursor-based. Shared `ListQuery = { cursor?: string, limit?: z.coerce.number().int().min(1).max(100).default(50), order?: "asc"|"desc" }`. Shared `Page<T> = { items: T[], nextCursor: string | null, total?: number }`.
- **Filtering:** list endpoints take typed filter query params (documented per resource).
- **Errors:** uniform `Problem` shape (RFC 9457 style) `{ type, title, status, detail?, code, fieldErrors? }`. Hono `onError` maps `ZodError → 422`, `AuthError → 401`, `CapabilityError → 403`, `NotFoundError → 404`, `ConflictError → 409`, `CycleError → 409` (dependency cycle), else `500`. `c.req.valid` validation failures use the `validator` hook to emit the same `Problem` shape with `fieldErrors`.
- **Idempotency:** all `POST` create routes accept optional `Idempotency-Key` header (validated, stored 24h) to dedupe retries.
- **Soft delete:** "delete" routes set `archived_at` (DELETE method, returns the archived row); hard delete is admin-only.
- **Provenance:** read endpoints surface `provenance { source, source_integration_id?, external_id?, external_url?, sync_mode }` for linked entities; `linked` rows are **read-only for mutating routes** except `link_external`/`unlink_external`.
- **Actor attribution:** every mutation records `created_by`/`actor_id` from `actorCtx`; agent-initiated calls (via MCP) additionally carry `initiator_id` (principal vs initiator) into the audit event.

## 2. Capability model (the `x-docket-capability` column)

Capabilities come straight from the data model: `{ view, comment, contribute, assign, manage }`. The table below uses these literals. Cross-org Hub routes require **no per-resource capability** beyond `authenticated` (they aggregate only over orgs where the user already has an active Actor, and each underlying item is capability-filtered server-side). Service-admin routes require a **`StaffUser` role** (`support | finance | superadmin`), shown as `staff:<role>`.

| Auth literal              | Meaning                                                                            |
| ------------------------- | ---------------------------------------------------------------------------------- |
| `public`                  | No session required (discovery, callbacks).                                        |
| `authenticated`           | Valid session; no org context.                                                     |
| `org:view` … `org:manage` | Org-scoped Actor + the named capability on the target resource (cascade-resolved). |
| `staff:<role>`            | A `StaffUser` row with ≥ the named role.                                           |

## 6. Validation, OpenAPI & RPC interplay (the one rule that must not be broken)

1. Define schema in `@docket/types`.
2. In the handler chain, attach **`validator("json"|"query"|"param", Schema)`** (input) and **`describeRoute({ responses: { 2xx: { content: { "application/json": { schema: resolver(OutSchema) } } } }, security, tags })`** (output + docs) — in that order, before the handler.
3. Return through `ok(c, OutSchema, data)` so runtime output matches the documented schema.
4. **Never** convert any router to `OpenAPIHono`/`createRoute` — it breaks the `.route()` chaining that the `AppType` export depends on. `hono-openapi`'s `validator` is the only path that satisfies _both_ RPC inference _and_ OpenAPI generation.
5. `openAPIRouteHandler(app, ...)` walks the same chained app to emit the spec; `servers` and `security` come from `@docket/env`.
